# v0.5 Human-Confirmed Publishing Workflow Release Candidate Checklist

本文档用于 v0.5 local fake/manual publishing workflow 的发布候选验收与 tag 准备。v0.5 Final validation 只做 release readiness 文档收口和本地质量门禁确认，不新增业务功能、不新增 API、不新增数据库表、不创建 Git tag、不 push、不 merge main。

## Release Scope

v0.5 的范围是建立 Human-Confirmed Publishing Workflow 的本地领域基础和 fake workflow 验证路径。当前支持：

- Publishing provider boundary documentation foundation。
- `PublishIntent` backend foundation。
- `PublicationRecord` backend foundation。
- `PublishIntent` confirm workflow。
- Deterministic `FakePublisherProvider` local execution。
- 项目详情页 Publishing / Fake Publishing workflow UI。
- Release Candidate 文档、测试边界和质量门禁收口。

当前 v0.5 本地 fake publishing 主链路：

```text
approved ReviewDraft
-> create PublishIntent
-> confirm PublishIntent
-> create PublicationRecord not_started
-> fake publish
-> PublicationRecord succeeded
-> frontend 项目详情页可操作
```

## Completed Batches

- Batch 1：Human-Confirmed Publishing Provider Boundary documentation foundation。
- Batch 2：PublishIntent / PublicationRecord backend domain foundation。
- Batch 3：PublishIntent confirm backend workflow foundation。
- Batch 4：PublisherProvider fake execution backend foundation。
- Batch 5：Publishing frontend workflow foundation。
- Batch 6：Publishing workflow Release Candidate stabilization / checklist。
- Final validation：Release validation / tag preparation 文档收口。

## Local Validation Commands

从仓库根目录执行：

```powershell
cd .\backend
.\.venv\Scripts\python.exe -m pytest

cd ..\frontend
npm.cmd run test -- --run
npm.cmd run build

cd ..
git diff --check
git status --short
```

验收时还应执行安全扫描，确认没有真实密钥、token、API key、secret、私钥、本机绝对路径、SQLite DB、`uploads/`、`node_modules/`、`.venv/`、`dist/`、生成媒体或运行时 preview artifacts 进入 Git。

## Acceptance Boundaries

- `Review Draft` approved 只表示草稿通过审核，不等于发布、不上传、不排期发布。
- `Review Draft` approved 不会自动创建 `PublishIntent`。
- `PublishIntent` 必须由用户显式创建，且只能基于同一项目内已 approved 的 Review Draft。
- `PublishIntent` confirmed 只表示用户确认进入发布执行准备阶段，不等于真实发布。
- confirm 只创建本地 `not_started` 的 `PublicationRecord` placeholder，不执行外部动作。
- Fake Publish 只调用 deterministic `FakePublisherProvider`，将本地 `PublicationRecord` 更新为 `succeeded` 并写入 fake external publication id。
- `PublicationRecord` succeeded 只表示本地 fake execution succeeded，不代表真实平台发布成功。
- archived project 可以读取已有 publish intents / publication records，但禁止 create / confirm / cancel / fake publish。
- cross-project publish intent / publication record 访问必须返回 404。
- v0.4 local fake/manual workflow 和 v0.3 render/subtitle/preview workflow 保持不变。

## Explicit Non-Goals

- 不实现真实 Douyin OAuth。
- 不实现真实上传。
- 不实现真实发布。
- 不保存 access token、refresh token、API key、secret 或任何凭据。
- 不调用真实 Douyin API 或任何外部服务。
- 不新增真实 PublisherProvider。
- 不实现自动发布。
- 不实现排期发布。
- 不生成真实媒体文件。
- 不接 `FFmpeg`、TTS、OpenAI、Claude、Gemini 或任何真实 AI Provider。
- 不把 fake succeeded 描述成真实平台发布成功。
- 不实现 v0.6 Metrics Feedback Loop。

## Security Boundary

v0.5 只写入本地 SQLite metadata 和 Git 跟踪的代码/文档。测试或本地运行产生的 SQLite 数据库、上传文件、runtime preview manifest、生成媒体或其他运行时输出只能位于 Git 忽略路径中，不能进入 Git。仓库不得提交密钥、token、私有账号信息、本机绝对路径、生成媒体、上传素材或私有数据。

## Tag Preparation Note

本清单完成后，分支可以准备进入 main 合并和后续 tag 流程。当前批次不创建 Git tag、不 push、不 merge main；tag 名称、tag message、push 与 merge 操作需由后续明确指令触发。
