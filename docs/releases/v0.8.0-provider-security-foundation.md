# v0.8.0 Provider & Credential Security Foundation

Release date: 2026-05-28

Previous stable release: `v0.7.0`

Tag: `v0.8.0`

Next planned stage: v0.9 Douyin Provider POC / Sandbox Integration

## Release Scope

v0.8.0 releases the Provider & Credential Security Foundation. It promotes v0.8 Batch 1-16 from release-candidate audit to the current stable release.

This release establishes provider/source separation, metadata-only backend security boundaries, read-only provider metadata APIs, frontend read-only security panels, secret redaction support, token lifecycle boundary metadata, and provider integration readiness summary metadata.

v0.8.0 does not include real Douyin integration, OAuth implementation, token storage, Credential storage, real metrics fetching, upload, publishing, scheduling, external service calls, or a production readiness claim.

- `GET /api/provider-oauth-boundaries/{provider_id}`

- `GET /api/provider-token-lifecycle-boundaries/{provider_id}`
- `GET /api/provider-readiness-summaries`

These APIs are read-only metadata surfaces. v0.8.0 adds no connect, authorize, callback, token exchange, refresh, revoke, disconnect, credential write, token read, upload, publish, schedule, readiness approval, readiness override, certification, or v0.9 POC start route.

## Metadata-only DB Tables

- `provider_connection_states`
- `provider_credential_references`
- `provider_security_audit_events`
- `provider_oauth_boundaries`
- `provider_token_lifecycle_boundaries`

Provider Integration Readiness Summary is computed from existing non-sensitive metadata and does not add a readiness persistence table.

These tables store provider/source/status/policy/safe-message/boundary-note style metadata only. They must not store access tokens, refresh tokens, token values, API keys, secrets, client secrets, authorization codes, OAuth client secrets, OAuth state values, credential material, encrypted credential material, private keys, raw requests, raw responses, raw payloads, token expiry values, token refresh responses, token revoke responses, or provider token responses.

## Security Boundaries

- `fake_local`, `douyin_sandbox`, and `douyin_real` remain explicitly separated.
- `fake_local` is local fake/demo/test workflow only.
- `douyin_sandbox` is sandbox placeholder metadata only.
- `douyin_real` is future real provider placeholder metadata only.
- Provider Registry is the provider source of truth.
- Credential references are metadata only and are not Credential storage.
- Secret redaction reduces leakage risk but is not a secret manager, KMS, credential vault, or encrypted token storage.
- OAuth Boundary metadata does not implement OAuth, callback routes, state storage, token exchange, or authorization URL generation.
- Token Lifecycle Boundary metadata does not store or refresh tokens, revoke tokens, disconnect providers, or rotate tokens.
- Integration Readiness Summary is a review aid, not production readiness certification.

## Release Verification Checklist

- Backend tests: `cd .\backend; .\.venv\Scripts\python.exe -m pytest`
- Frontend tests: `cd .\frontend; npm.cmd run test -- --run`
- Frontend build: `cd .\frontend; npm.cmd run build`
- Whitespace check: `git diff --check`
- Git status review: `git status --short`
- Security scan confirms no real token, refresh token, secret, API key, credential, authorization code, OAuth client secret, OAuth state value, SQLite DB, uploads, dist, node_modules, `.venv`, generated media, runtime file, real provider response, real Douyin credential, or real OAuth callback credential enters Git.
