# v1.0 Douyin Integration User Test Release Readiness Checklist

v1.0 is the Douyin Integration User Test Release target. As of Batch 2, it is in OAuth state storage / anti-replay foundation work for small-scope user testing and is not a production commercial release.

Batch 2 adds internal-only OAuth state foundation code and a digest-only database table. It does not implement real OAuth runtime behavior, OAuth URLs, OAuth start routes, OAuth callback routes, token exchange, token storage, credential storage, frontend OAuth UI, Provider runtime behavior, real Douyin API calls, real metrics reads, uploads, publishing, or scheduled publishing.

## Batch 0 Planning Readiness

- [ ] ADR 0045 exists and is accepted for v1.0 Batch 0 planning.
- [ ] `docs/plans/v1.0-douyin-user-test-release-plan.md` exists.
- [ ] README and README.en state that v0.9.0 has been released as Douyin Provider POC / Sandbox Integration.
- [ ] README and README.en state that v1.0 has advanced to Batch 2 OAuth state storage / anti-replay foundation but real OAuth runtime implementation has not started.
- [ ] Roadmap, architecture, product spec, development docs, and v1-to-v2 roadmap distinguish v1.0 from v1.5 and v2.0.
- [ ] Documentation states that v1.0 is not a production commercial release.
- [ ] Documentation states that Batch 0 does not implement real OAuth, token lifecycle, credential storage, publish, metrics, API, DB, Provider, or frontend OAuth UI behavior.

## Batch 1 OAuth Boundary / Callback Contract Readiness

- [ ] ADR 0046 exists and is accepted for v1.0 Batch 1 OAuth boundary / callback contract.
- [ ] `docs/contracts/v1.0-douyin-oauth-callback-contract.md` exists and states that it is a future contract only, not a current route.
- [ ] `docs/testing/v1.0-oauth-callback-contract-test-matrix.md` exists and uses contract scenarios only, with no real service calls.
- [ ] Documentation covers authorization start, success callback, error callback, unsupported provider, cancelled authorization, malformed callback, missing state, replayed callback, and expired callback categories.
- [ ] Documentation maps future safe response categories to `pending_token_exchange`, `authorization_cancelled`, `authorization_failed`, `invalid_callback`, `unsupported_provider`, and `oauth_temporarily_disabled`.
- [ ] Documentation states that future callback routes must rely on Batch 2 OAuth state storage / anti-replay foundation and still require a later accepted route batch.
- [ ] Documentation states that token exchange must wait for Batch 3.
- [ ] Documentation states that credential storage must wait for Batch 4.
- [ ] Documentation states that real OAuth runtime enablement must wait for Batch 5 feature flag / kill switch.
- [ ] Documentation states that `fake_local`, `douyin_sandbox`, and `douyin_real` remain strictly separated.
- [ ] Documentation states that `douyin_real` must not fallback to `douyin_sandbox`.
- [ ] Documentation states that logs, audit records, error responses, and fixtures must redact token, secret, credential, authorization code, OAuth state, cookie, session, raw request, and raw response values.
- [ ] Documentation states that Batch 1 does not implement real OAuth, OAuth URLs, callback routes, state storage, token exchange, credential storage, API, DB, Provider runtime behavior, frontend OAuth UI, external service calls, publishing, or metrics.

## Batch 2 OAuth State Storage / Anti-Replay Foundation Readiness

- [ ] ADR 0047 exists and is accepted for v1.0 Batch 2 OAuth state storage / anti-replay foundation.
- [ ] `docs/contracts/v1.0-douyin-oauth-state-storage-contract.md` exists and states that the state foundation is internal-only, not a public API contract.
- [ ] `docs/testing/v1.0-oauth-state-storage-test-matrix.md` exists and covers create, digest-only persistence, one-time consume, replay reject, expiry reject, provider mismatch, unsupported provider, source separation, redaction, no external service, no OAuth route, no token exchange, and no credential storage expectations.
- [ ] `provider_oauth_states` exists as a digest-only schema and does not include raw state, state value, authorization code, access token, refresh token, token, secret, credential, cookie, session, raw request, or raw response columns.
- [ ] OAuth state creation stores only digest and safe metadata.
- [ ] Anti-replay tests prove that pending state can be consumed once and consumed state cannot be reused.
- [ ] Expiry tests prove expired state is rejected and safely marked.
- [ ] Missing and malformed state tests safely fail without leaking inputs.
- [ ] Provider mismatch and unsupported provider tests safely fail without sandbox fallback.
- [ ] Redaction tests prove safe messages and fixtures do not contain real token, secret, credential, authorization code, raw OAuth state, OAuth state value, cookie, session, API key, raw request, or raw response values.
- [ ] Tests verify no OAuth start route, OAuth callback route, token exchange route, token storage, credential storage, frontend OAuth UI, Douyin API call, upload, publish, scheduling, or real metrics read exists in Batch 2.

## Platform Preconditions

- [ ] Douyin Open Platform app readiness is confirmed.
- [ ] App review / approval requirements are documented.
- [ ] OAuth permission scopes are confirmed.
- [ ] Callback URL requirements are confirmed.
- [ ] Token lifecycle policy is accepted.
- [ ] Credential encryption / storage design is accepted.
- [ ] User authorization consent design is accepted.
- [ ] Platform rate limit / error policy is documented.
- [ ] Audit log design is accepted.
- [ ] Kill switch / feature flag design is accepted.

## OAuth Readiness

- [ ] Real Douyin OAuth flow has an accepted ADR before implementation.
- [ ] OAuth boundary / callback contract has accepted ADR 0046.
- [ ] User authorization is explicit and documented.
- [ ] OAuth consent boundaries and platform permission requirements are documented.
- [ ] OAuth flow can be disabled by feature flag or kill switch.

## Callback Readiness

- [ ] OAuth callback route has an accepted ADR before implementation.
- [ ] Callback validation rejects malformed, replayed, and unsupported provider responses.
- [ ] Callback logging is redacted.
- [ ] Callback tests do not use real authorization values.

## OAuth State Readiness

- [ ] OAuth state storage has accepted ADR 0047 before any callback route implementation.
- [ ] Replay protection is tested with one-time consume behavior.
- [ ] State expiry and cleanup / marking behavior are documented.
- [ ] Raw state values are never persisted or exposed in user-facing responses, logs, audit records, fixtures, or examples.

## Token Lifecycle Readiness

- [ ] Token exchange has an accepted ADR before implementation.
- [ ] Token refresh, expiry, revoke, and disconnect behavior are documented.
- [ ] Token lifecycle tests use fake values only.
- [ ] Token lifecycle errors have safe user-facing messages.

## Credential Storage Readiness

- [ ] Encrypted credential storage has an accepted ADR before any real token storage.
- [ ] Business tables store only credential references, not credential material.
- [ ] Secret redaction covers logs, audits, tests, and responses.
- [ ] Backup and restore handling for encrypted credentials is documented.

## Real Provider Isolation

- [ ] `fake_local`, `douyin_sandbox`, and `douyin_real` remain separated.
- [ ] Sandbox data is never shown as real Douyin data.
- [ ] Real provider operations cannot fallback to sandbox silently.
- [ ] Unknown provider ids fail safely.

## User-Confirmed Publish Readiness

- [ ] Publish workflow requires explicit user confirmation.
- [ ] Publish intent is linked to review state.
- [ ] Publish preflight validation is tested.
- [ ] Duplicate publish prevention is documented and tested.

## Metrics Permission Readiness

- [ ] Real metrics read has an accepted ADR before implementation.
- [ ] Metrics permissions and platform limitations are documented.
- [ ] Metrics freshness is visible to testers.
- [ ] Metrics responses are redacted before logging.

## Audit Log Readiness

- [ ] OAuth, connection, publish, metrics, revoke, and disconnect events are audit-covered.
- [ ] Audit entries contain no sensitive material.
- [ ] Audit tests verify redaction.
- [ ] User-facing support flows can reference safe audit metadata.

## Kill Switch / Feature Flag Readiness

- [ ] Real Douyin integration can be disabled quickly.
- [ ] Publish and metrics features can be disabled independently.
- [ ] Disablement behavior is tested.
- [ ] User messaging for disabled features is documented.

## Security Scan Readiness

- [ ] Tests verify no real token, secret, API key, credential, authorization code, OAuth state, cookie, or session values are committed.
- [ ] Generated files, SQLite DB files, uploads, dist, node_modules, `.venv`, and runtime artifacts are excluded.
- [ ] Dependency and static checks are documented for the user test release.

## Documentation Readiness

- [ ] User test scope is documented.
- [ ] Platform permission, app review, API limit, and failure mode docs are available.
- [ ] Rollback and disablement docs are available.
- [ ] The docs clearly state that v1.0 is not a production commercial release.

## User Test Acceptance Criteria

- [ ] A small set of test users can connect with explicit authorization.
- [ ] A user-confirmed publish path can be validated under test constraints.
- [ ] Publish status can be tracked safely.
- [ ] Minimum metrics read can be tested when permissions allow.
- [ ] Failures are observable, recoverable, and redacted.

## Rollback / Disablement Criteria

- [ ] Any credential leakage risk stops the user test.
- [ ] Any platform policy violation risk stops the user test.
- [ ] Any uncontrolled publish risk stops the user test.
- [ ] Any missing disablement path stops the user test.
- [ ] Any sandbox data presented as real data stops the user test.

## Commercial Boundary

- [ ] v1.0 is not a production commercial release.
- [ ] v1.0 does not provide SLA commitments.
- [ ] v1.0 does not provide multi-tenant SaaS.
- [ ] v1.0 does not onboard customer-of-customer end users.
