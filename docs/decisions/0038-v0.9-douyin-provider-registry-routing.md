# ADR 0038: v0.9 Douyin Provider Registry / Factory Routing

Status: Accepted

## Context

v0.9 Batch 1 introduced backend-only Douyin Provider Adapter Skeleton classes for `douyin_sandbox` and `douyin_real`. v0.9 Batch 2 added deterministic sandbox-only operation simulation for `douyin_sandbox` while keeping `douyin_real` blocked / not implemented.

The next safe backend step is to decouple provider selection from adapter implementation. Later sandbox/mock connection, metrics POC, and real-provider boundary work need a stable way to resolve a known provider id into safe metadata and the matching adapter without introducing API routes, OAuth, token storage, database changes, frontend UI, or external calls.

## Decision

- This batch is v0.9 Batch 3.
- This batch introduces a backend-only Douyin Provider Registry / Factory Routing foundation.
- The registry defines local descriptors for the whitelisted provider ids `douyin_sandbox` and `douyin_real`.
- The factory creates the matching adapter for an exact whitelisted provider id.
- `douyin_sandbox` continues to use sandbox-only deterministic simulation.
- `douyin_real` remains blocked / not implemented.
- Unknown provider ids fail explicitly and must not fallback to sandbox.
- No external service calls are allowed.
- No OAuth implementation is introduced.
- No OAuth URL generation is introduced.
- No callback route is introduced.
- No token exchange or storage is introduced.
- No backend API, database schema change, or frontend UI is introduced.

## Registry Boundary

The registry is a backend-only local metadata layer. It may list descriptors, return a descriptor for a whitelisted provider id, and route exact provider ids to adapter instances.

Descriptors may include safe fields such as provider id, display name, environment, status, simulation support, real OAuth support, real publish support, and real metrics support. Descriptors must not include fields intended to carry credentials, OAuth material, cookies, sessions, raw platform payloads, or real platform responses.

The registry must not read environment variables, access the database, create HTTP clients, load platform SDKs, create OAuth URLs, perform token exchange, or save credentials.

## Factory Boundary

The factory creates `DouyinSandboxAdapter` only for `douyin_sandbox` and `DouyinRealAdapter` only for `douyin_real`.

The factory must reject unknown provider ids, empty strings, missing ids, case variants, and whitespace variants unless a future ADR defines a normalization policy. Rejection must be explicit and must not fallback to sandbox.

## Sandbox Boundary

`douyin_sandbox` routed through the factory remains sandbox-only. It may return deterministic dry-run simulation results with stable fake references, but it must not represent real OAuth, real callback handling, real token exchange, real metrics fetching, upload, publish, scheduling, or any real Douyin API behavior.

## Real Provider Boundary

`douyin_real` routed through the factory remains a future real provider placeholder. It must continue returning blocked / not implemented operation results and must not return success, read configuration, create authorization URLs, perform token exchange, fetch metrics, upload, publish, schedule, or call external services.

## Operation Routing Boundary

Routing through the factory must preserve the Batch 2 adapter operation contract:

- `douyin_sandbox` operations remain deterministic sandbox-only simulation.
- `douyin_real` operations remain blocked / not implemented.
- Operation result shapes remain compatible with existing adapter skeleton and sandbox operation tests.
- No routed operation may read credentials, read or write tokens, call external services, or persist data.

## Source Separation

`fake_local`, `douyin_sandbox`, and `douyin_real` remain separate:

- `fake_local` remains local fake/demo/test workflow only.
- `douyin_sandbox` remains sandbox/mock simulation only.
- `douyin_real` remains future real provider behavior only.

Sandbox results must not be shown or treated as real Douyin data. Unknown provider ids must not be reinterpreted as sandbox.

## Security Requirements

- Do not call real Douyin APIs.
- Do not call external HTTP or SDK services.
- Do not read environment secrets.
- Do not read credentials.
- Do not read or write tokens.
- Do not save authorization codes or OAuth state.
- Do not create OAuth URLs.
- Do not add backend API routes.
- Do not modify database tables or migrations.
- Do not add frontend UI.
- Keep errors safe and avoid leaking environment values.
- Keep tests explicit about route absence, source separation, unknown-provider failure, no fallback, no network calls, and no sensitive value leakage.

## Non-Goals

- 不接真实 Douyin API。
- 不实现 OAuth。
- 不创建 OAuth URL。
- 不新增 OAuth callback route。
- 不新增 OAuth state storage。
- 不新增 token exchange。
- 不新增 token storage。
- 不新增 credential storage。
- 不保存 token。
- 不保存 secret。
- 不保存 API key。
- 不保存 credential。
- 不保存 authorization code。
- 不保存 OAuth state。
- 不读取真实环境变量密钥。
- 不抓取真实指标。
- 不上传。
- 不发布。
- 不排期发布。
- 不调用外部服务。
- 不新增 backend API。
- 不修改数据库表。
- 不新增前端 UI。
- 不声明 v0.9 POC 已完成。

## Consequences

Batch 3 gives later workflows a stable provider-selection entry point without enabling real platform behavior. Tests can now verify that exact provider ids route to the expected adapter, unknown providers do not fallback, sandbox behavior remains deterministic, and real-provider behavior remains blocked.

This decision does not make v0.9 POC complete. It only establishes safe registry / factory routing for later sandbox/mock account connection, provider status dry-run wiring, sandbox metrics POC, and real-provider boundary design.
