# ADR 0050: v1.0 Real Provider Feature Flag / Kill Switch

## Status

Accepted for v1.0 Batch 5 planning and internal foundation.

## Context

v1.0 is the Douyin Integration User Test Release target, but the repository still must not enable real Douyin runtime behavior. Batch 1 defined the OAuth callback contract, Batch 2 added internal OAuth state storage / anti-replay foundation, Batch 3 added an internal token exchange boundary with fake-gated simulation, and Batch 4 added an internal credential reference / storage boundary.

Batch 5 establishes the real provider feature flag / kill switch foundation that later real-provider batches must pass before any runtime action is considered. This is a safety gate, not a real-provider enablement.

## Decision

Batch 5 adds an internal-only real provider controls boundary. It may add a backend service and tests that evaluate known provider, source type, feature flag status, kill switch status, requested capability, environment status, platform preconditions, and sandbox fallback attempts. The decision result must be safe metadata only.

Batch 5 does not enable the real provider. `douyin_real` remains disabled / blocked by default. Kill switch behavior is safe by default: unknown, missing, malformed, disabled, revoked, expired, or policy-denied control states must block real provider behavior. A real provider operation must satisfy known provider, `source_type=real`, explicitly enabled feature flag, inactive kill switch, explicitly allowed capability, allowed environment, and accepted platform preconditions before later batches can even consider runtime behavior. In Batch 5, even fully accepting control metadata still returns a blocked real-provider decision because real runtime enablement is not part of this batch.

`douyin_real` must not fallback to `douyin_sandbox`. `fake_local`, `douyin_sandbox`, and `douyin_real` remain strictly separated.

## Explicit Non-Goals

Batch 5 does not:

- Enable a real provider runtime.
- Implement real OAuth.
- Add OAuth start routes.
- Add OAuth callback routes.
- Create OAuth URLs.
- Implement real token exchange.
- Save real tokens, secrets, or credentials.
- Save authorization code, raw OAuth state, cookie, session, API key, bearer, raw request, or raw response material.
- Call Douyin API or any business external service.
- Add frontend OAuth UI.
- Upload, publish, schedule publishing, or read real metrics.
- Modify the `v0.9.0` tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Declare production, commercial, or SaaS readiness.

## Safe Metadata Rules

Feature flag / kill switch results may include provider id, source type, requested capability, feature flag status, kill switch status, environment status, platform preconditions status, decision category, booleans that confirm no real provider or external service was enabled, and a safe status message.

Feature flag / kill switch results must not include token, access token, refresh token, secret, client secret, credential, authorization code, raw OAuth state, OAuth state value, cookie, session, API key, bearer, raw request, or raw response values.

All logs, audit records, responses, tests, fixtures, and documentation examples must stay redacted. Tests may use obvious fake values only for negative assertions.

## Dependency Gates

- Batch 2 remains required before any state handling.
- Batch 3 remains required before token exchange boundary behavior.
- Batch 4 remains required before credential reference boundary behavior.
- Batch 5 only provides feature flag / kill switch foundation.
- Future real OAuth, publish, metrics, and provider runtime behavior still require separate ADRs, tests, security scans, platform permissions, app review, user authorization, and accepted preconditions.

## Consequences

Later real-provider work must call this boundary before attempting real OAuth, token exchange, credential storage, publishing, publish status reconciliation, metrics read, disconnect, or revoke behavior.

Any missing, malformed, unknown, disabled, revoked, expired, or policy-denied control state must stop at a safe blocked result. This protects user testing from accidental real-provider enablement and keeps sandbox behavior from being mistaken for real Douyin behavior.
