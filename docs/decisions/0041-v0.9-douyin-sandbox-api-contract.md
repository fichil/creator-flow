# ADR 0041: v0.9 Douyin Sandbox API Contract

Status: Accepted

## Context

v0.9 Batch 1 introduced the backend-only Douyin adapter skeleton. Batch 2 added deterministic sandbox-only operations for `douyin_sandbox`. Batch 3 added provider registry / factory routing. Batch 4 added the backend-only sandbox metrics / mock workflow POC. Batch 5 aligned the roadmap to v1.0, v1.5, and v2.0.

The next safe step is to expose a minimal backend API contract that later frontend sandbox POC or smoke checks can call. The API must remain sandbox-only and must not be confused with real Douyin, OAuth, token, metrics, publish, production, or SaaS capability.

## Decision

v0.9 Batch 6 introduces backend-only Douyin sandbox API contract / smoke endpoints. These endpoints are a v0.9 POC callable boundary, not production API.

The API may use the Batch 3 registry / factory route and Batch 4 sandbox workflow helpers to call `douyin_sandbox`. It must return deterministic sandbox / simulated / dry-run results only.

`douyin_real` remains blocked / not implemented through this API. Unknown providers fail explicitly and must not fallback to sandbox.

## API Boundary

The sandbox API may expose:

- A Douyin provider descriptor list.
- A Douyin provider descriptor lookup for `douyin_sandbox` and `douyin_real`.
- A sandbox mock account connection endpoint.
- A sandbox metrics preview endpoint.
- A sandbox publish dry-run endpoint.

All endpoint responses must be deterministic, explicitly sandbox-labeled, and free of sensitive payload fields. The API does not read or write database state and does not create real platform side effects.

## Sandbox Boundary

`douyin_sandbox` endpoint results may show stable fake ids and deterministic payloads such as sandbox connection, sandbox metrics, and sandbox publish references. These results are only contract-test payloads. They are not real Douyin account connections, real OAuth, real metrics, real uploads, real publishes, or scheduled publishing.

## Real Provider Boundary

`douyin_real` can be identified by descriptor, but operation endpoints must return blocked / not implemented responses. They must not return success, create authorization URLs, read configuration, exchange tokens, fetch metrics, upload, publish, schedule, or call external services.

## Unknown Provider Boundary

Unknown provider ids must return an explicit unsupported-provider response. They must not fallback to `douyin_sandbox`, must not hint that a real token or secret should be supplied, and must not expose sensitive-looking provider id content.

## Non-Goals

This batch does not:

- Connect to real Douyin.
- Implement OAuth.
- Create OAuth URLs.
- Add OAuth callback routes.
- Add OAuth state storage.
- Add token exchange.
- Store token, secret, API key, credential, authorization code, or OAuth state values.
- Read real environment secrets.
- Call external services.
- Fetch real metrics.
- Upload, publish, or schedule publishing.
- Modify database tables or migrations.
- Add frontend UI.
- Add real tenant, billing, RBAC, or admin console implementation.
- Declare the v0.9 POC complete.
- Declare v1.0, v1.5, or v2.0 complete.

## Consequences

Batch 6 gives later frontend sandbox POC work and smoke-api checks a stable backend callable surface while preserving the sandbox-only boundary. It makes the POC easier to test without enabling real platform behavior.

This decision does not make v0.9 production-ready, commercial-ready, real Douyin-ready, OAuth-ready, token-ready, metrics-ready, publish-ready, or SaaS-ready.

## Follow-up Work

- Add frontend sandbox POC calls in a separate batch if approved.
- Extend smoke-api coverage in a separate batch if needed.
- Define v0.9 release readiness separately.
- Define any real Douyin, OAuth, token, metrics, upload, publish, or scheduling API in separate v1.0+ ADRs and implementation batches.
