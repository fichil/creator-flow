# ADR 0042: v0.9 Douyin Frontend Sandbox POC Panel

Status: Accepted

## Context

v0.9 Batch 6 introduced a backend-only Douyin sandbox API contract / smoke endpoint surface. That API can return deterministic provider descriptors, sandbox mock connection results, sandbox metrics preview results, and sandbox publish dry-run results without connecting to real Douyin.

The next safe step is to add a frontend panel that can call only those sandbox endpoints and make the POC boundary visible to reviewers. The panel must not be confused with real Douyin account connection, OAuth, token, metrics, upload, publish, production, or commercial capability.

## Decision

v0.9 Batch 7 introduces a frontend Douyin Sandbox POC Panel.

The panel:

- Calls only the v0.9 Batch 6 sandbox-only backend API.
- Displays Douyin provider descriptors from the sandbox API.
- Shows `douyin_sandbox` as sandbox / simulated / dry-run only.
- Shows `douyin_real` as blocked / not implemented.
- Allows sandbox-only calls for mock connection, metrics preview, and publish dry-run.
- Displays deterministic sandbox / simulated / dry-run results only.
- Does not add real OAuth UI, real token input, real platform key input, real upload, real publish, or scheduling controls.

Unknown provider ids continue to fail in the backend and must not fallback to `douyin_sandbox`.

## Frontend Boundary

The frontend panel is a visual sandbox smoke surface for v0.9 POC review. It is not a production UI. It must clearly state that all results are deterministic simulated / dry-run responses and that the panel does not connect to real Douyin, perform OAuth, store tokens, upload videos, publish content, or schedule content.

The frontend API client must not hard-code Douyin domains, must not send authorization headers, must not read token / secret / credential environment variables, and must be tested with mocked fetch calls.

## Sandbox API Boundary

The panel may call:

- `GET /api/providers/douyin`
- `POST /api/providers/douyin/sandbox/mock-connection`
- `POST /api/providers/douyin/sandbox/metrics-preview`
- `POST /api/providers/douyin/sandbox/publish-dry-run`

It may render deterministic fake ids such as sandbox connection, sandbox metrics, sandbox video, and sandbox publish references. These are contract-test payloads only and must not be presented as real Douyin account connection, real metrics, real upload, real publish, or real scheduling.

## Real Provider Boundary

`douyin_real` remains visible only for source separation and must remain blocked / not implemented. The panel must not offer a real connect button, real OAuth button, token viewer, platform key form, upload control, publish button, or scheduling control for `douyin_real`.

## Non-Goals

This batch does not:

- Connect to real Douyin.
- Implement OAuth.
- Create OAuth URLs.
- Add OAuth callback routes.
- Add OAuth state storage.
- Add token exchange.
- Store token, secret, API key, credential, authorization code, or OAuth state values.
- Read real environment secrets.
- Call Douyin API or third-party services.
- Fetch real metrics.
- Upload, publish, or schedule publishing.
- Modify database tables or migrations.
- Add real tenant, billing, RBAC, or admin console implementation.
- Declare the v0.9 POC complete.
- Declare v1.0, v1.5, or v2.0 complete.

## Consequences

Batch 7 gives the v0.9 POC a visible frontend sandbox smoke surface while preserving the sandbox-only boundary. Reviewers can exercise the Batch 6 callable surface through UI without enabling any real platform behavior.

This decision does not make v0.9 production-ready, commercial-ready, real Douyin-ready, OAuth-ready, token-ready, metrics-ready, publish-ready, or SaaS-ready.

## Follow-up Work

- Use Batch 8 for POC readiness finalization / release candidate work if approved.
- Keep any real Douyin, OAuth, token, metrics, upload, publish, or scheduling UI in separate v1.0+ ADRs and implementation batches.
- Continue sensitive payload tests and sandbox boundary tests for any provider UI.
