# ADR 0054: v1.0 Limited Metrics Read Guarded Foundation

## Status

Accepted for v1.0 Batch 9 limited metrics read guarded foundation.

## Context

v1.0 is still the Douyin Integration User Test Release target, not a production, commercial, or SaaS-ready release. Batch 8 added local-only, metadata-only publish status reconciliation and status snapshots, but it did not query real Douyin status or open any real metrics path.

Batch 9 establishes the guarded foundation for future limited real metrics read. It is a safety boundary, data-shape layer, freshness-visibility layer, and user-test preparation layer only. It does not provide real Douyin metrics read by default.

## Decision

Batch 9 adds metadata-only metrics snapshot records and a local-only metrics snapshot API / UI. Every metrics snapshot must depend on an existing Batch 8 publish status snapshot, which in turn depends on a Batch 7 publish attempt and Batch 6 publish intent. The service must validate the chain through the snapshot, reject missing or non-eligible publish status, validate provider identity, check Batch 5 real provider controls for real providers, reject unsupported providers, reject sandbox fallback attempts, reject external metrics query requests by default, represent missing metrics permission safely, expose freshness metadata, reject malformed fake or sandbox-safe fixtures, and return safe result metadata only.

Batch 9 does not call the real Douyin metrics API by default. It does not call Douyin API or any business external service. It does not upload video, publish, schedule publishing, query real Douyin publication status, create OAuth URLs, add OAuth start routes, add OAuth callback routes, perform real token exchange, or store token / secret / credential / authorization code / raw OAuth state material.

Metrics snapshots are metadata-only. They may store local safe counts, basis-point completion rate, metrics source, freshness status, observed timestamp, external query status, safe messages, and change reasons. They must not persist, return, log, audit, or place in fixtures raw metrics response, raw provider response, raw status response, raw external response, raw Douyin response, raw request, raw response, token, secret, credential, authorization code, cookie, session, API key, bearer, raw OAuth state, upload response, or publish response material.

Metrics freshness must be visible to users as `fresh`, `stale`, `unknown`, or `not_available`. Metrics permission and platform limitation states must also be visible as safe metadata. Fake-local and sandbox-safe fixtures may use obvious fake metric values, but those fixtures must not be represented as real Douyin metrics and must not be stored or returned as raw responses.

`douyin_real` remains blocked by default through Batch 5 real provider controls. `douyin_real` must not fallback to `douyin_sandbox`.

## Explicit Non-Goals

Batch 9 does not:

- Call the real Douyin metrics API.
- Call Douyin API or any business external service.
- Query real Douyin publication status.
- Upload video.
- Execute real publishing.
- Schedule publishing.
- Add OAuth start routes.
- Add OAuth callback routes.
- Create OAuth URLs.
- Implement real OAuth.
- Implement real token exchange.
- Store token, secret, credential, authorization code, raw OAuth state, cookie, session, API key, bearer, raw request, raw response, provider response, publish response, status response, metrics response, external response, or Douyin response material.
- Add real token storage or real credential storage.
- Enable `douyin_real`.
- Allow `douyin_real` fallback to `douyin_sandbox`.
- Modify the `v0.9.0` tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Declare production, commercial, or SaaS readiness.

## Safe Metrics Rules

- A publish status snapshot is required before metrics snapshot creation.
- The snapshot must belong to the same project and must resolve to a valid publish attempt and publish intent chain.
- Only eligible local publish status values can create metrics snapshots.
- External metrics query requests are blocked by default.
- Metrics permission metadata must exist, or the result must be `metrics_permission_missing`.
- Unknown freshness is represented explicitly and safely.
- Metrics source values are limited to `local`, `fake_fixture`, or `sandbox_fixture`.
- Real provider controls must be checked before any real-provider metrics foundation path is considered, and Batch 9 still blocks real runtime behavior.

## Dependency Gates

- Batch 8 publish status reconciliation / status snapshot is required before Batch 9 metrics snapshots.
- Batch 7 metadata-only publish attempt and Batch 6 publish intent remain required through the status snapshot chain.
- Batch 5 real provider controls continue to block `douyin_real`.
- Future real metrics read requires a separate accepted ADR, tests, security scan, platform permission confirmation, credential readiness, explicit user authorization, app review, rate-limit handling, audit / redaction coverage, and explicit runtime enablement.
- Batch 10 can package user test readiness / release candidate materials, but it must not treat Batch 9 as production readiness.

## Consequences

Future real metrics work can rely on a local metadata-only snapshot shape, freshness vocabulary, permission-limitation visibility, and redaction contract without implying that real Douyin metrics have been fetched. Batch 9 keeps metrics behavior reviewable and testable while preserving source separation, human-in-the-loop publishing, and the real-provider disabled boundary.
