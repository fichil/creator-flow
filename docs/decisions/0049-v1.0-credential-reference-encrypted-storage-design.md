# ADR 0049: v1.0 Credential Reference And Encrypted Storage Design

Status: Accepted

## Context

ADR 0047 added the Batch 2 OAuth state storage / anti-replay foundation. ADR 0048 added the Batch 3 internal token exchange boundary with fake-gated integration.

v1.0 Batch 4 establishes the credential reference and encrypted storage boundary needed before any future real token storage can be considered. This batch is a safety design and internal boundary step. It does not make real Douyin credential storage available.

## Decision

Batch 4 adds an internal-only credential storage boundary service, backend tests, and documentation. The boundary accepts only safe metadata from the Batch 3 token exchange boundary and can create metadata-only credential references. It does not accept raw token material, raw authorization code, raw OAuth state, secret, credential, cookie, session, bearer, raw request, or raw response values.

The existing `provider_credential_references` table remains metadata-only. Batch 4 does not add an encrypted payload table. Because the project does not currently have an accepted reliable encryption provider or key-management design, Batch 4 must not use base64, hashing, XOR, ordinary encoding, or a custom algorithm as pretend encryption. Any request that requires encrypted credential persistence returns a safe disabled / unavailable category.

Real credential persistence remains disabled until a later accepted real-provider enablement path exists.

## Security Boundary

Batch 4 does not:

- Add an OAuth start route.
- Add an OAuth callback route.
- Create an OAuth URL.
- Implement real OAuth.
- Implement real token exchange.
- Call Douyin APIs or business external services.
- Read real client secrets.
- Store real access tokens.
- Store real refresh tokens.
- Store token values.
- Store authorization codes.
- Store raw OAuth state.
- Store secrets.
- Store credential material.
- Store cookies or sessions.
- Add frontend OAuth UI.
- Enable `douyin_real`.

All logs, audit records, responses, tests, fixtures, and documentation examples must remain redacted. Obvious fake values may appear only in tests as negative assertions and must never be returned or persisted as credential material.

## Provider Isolation

`fake_local`, `douyin_sandbox`, and `douyin_real` remain strictly separated. `douyin_real` is blocked / disabled by default and must not fallback to `douyin_sandbox`.

## Dependency Gates

- Batch 2 state validation must exist before callback state handling.
- Batch 3 token exchange boundary must exist before credential reference handoff.
- Batch 4 only provides credential reference / encrypted storage boundary behavior.
- Batch 5 feature flag / kill switch controls must exist before real provider runtime enablement.

## Consequences

The project gains reviewable credential storage guardrails and tests for metadata-only reference creation, raw material rejection, real provider disablement, sandbox fallback prevention, encryption-provider unavailable behavior, redaction, and no external service calls.

Batch 4 does not make v1.0 complete, does not make v1.5 or v2.0 complete, and does not claim production, commercial, or SaaS readiness.
