# ADR 0053: v1.0 Publish Status Reconciliation

## Status

Accepted for v1.0 Batch 8 publish status reconciliation foundation.

## Context

v1.0 is still the Douyin Integration User Test Release target, not a production, commercial, or SaaS-ready release. Batch 7 added a metadata-only guarded publish attempt layer, but it did not execute real publishing or create any real provider status query path.

Batch 8 establishes the local publish status reconciliation foundation before any future real Douyin publication status query can be considered. It is a local safety and data-shape layer only.

## Decision

Batch 8 adds metadata-only publish status reconciliation and publish status snapshot records. Every reconciliation must depend on an existing Batch 7 publish attempt that belongs to the same project. The service must reject missing attempts, non-ready attempts, duplicate active reconciliation requests, unsupported providers, sandbox fallback attempts, real-provider blocked states, external status query requests, malformed fixtures, and stale status updates.

Batch 8 does not query real Douyin publication status by default. It does not call Douyin API or any business external service. It does not upload video, publish, schedule publishing, fetch metrics, create OAuth URLs, add OAuth start routes, add OAuth callback routes, perform real token exchange, or store token / secret / credential / authorization code / raw OAuth state material.

Status snapshots and reconciliation records are metadata-only. They may store local safe status categories, source type, fixture source, timestamps, safe messages, and change reasons. They must not persist, return, log, audit, or place in fixtures raw provider response, raw upload response, raw publish response, raw status response, raw external response, raw Douyin response, raw request, raw response, token, secret, credential, authorization code, cookie, session, API key, bearer, raw OAuth state, or metrics response material.

`douyin_real` remains blocked by default through Batch 5 real provider controls. `douyin_real` must not fallback to `douyin_sandbox`.

Fake-local and sandbox-safe reconciliation may use obvious fake fixture statuses, but those fixtures must be converted into safe local status metadata and must not be stored or returned as raw provider responses.

## Explicit Non-Goals

Batch 8 does not:

- Query real Douyin publication status.
- Call Douyin API or any business external service.
- Upload video.
- Execute real publishing.
- Schedule publishing.
- Fetch real metrics; limited real metrics read must wait for Batch 9.
- Add OAuth start routes.
- Add OAuth callback routes.
- Create OAuth URLs.
- Implement real OAuth.
- Implement real token exchange.
- Store token, secret, credential, authorization code, raw OAuth state, cookie, session, API key, bearer, raw request, raw response, provider response, upload response, publish response, status response, external response, Douyin response, or metrics response material.
- Add real token storage or real credential storage.
- Enable `douyin_real`.
- Allow `douyin_real` fallback to `douyin_sandbox`.
- Modify the `v0.9.0` tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Declare production, commercial, or SaaS readiness.

## Safe Reconciliation Rules

- A publish attempt is required before reconciliation.
- Reconciliation must validate the attempt belongs to the project.
- Blocked, cancelled, failed-safe, or guard-blocked attempts are not eligible by default.
- Duplicate active reconciliation for the same attempt is rejected.
- Stale local status updates are ignored safely.
- External status query requests are blocked.
- Safe local status values must remain local and metadata-only.
- Status source values are limited to `local`, `fake_fixture`, or `sandbox_fixture`.
- Real provider controls must be checked before any real-provider reconciliation is considered, and Batch 8 still blocks real runtime behavior.

## Dependency Gates

- Batch 7 metadata-only publish attempt is required before Batch 8 reconciliation.
- Batch 5 real provider controls continue to block `douyin_real`.
- Future real publish status query requires a separate accepted ADR, tests, security scan, platform preconditions, credential readiness, explicit user authorization, audit / redaction coverage, and explicit runtime enablement.
- Batch 9 is required before limited real metrics read.

## Consequences

Future real publishing work can rely on a local reconciliation and snapshot shape without implying that real platform status has been queried. Batch 8 keeps the system reviewable and testable while preserving the human-in-the-loop publishing boundary and real/fake/sandbox source separation.
