# ADR 0047: v1.0 OAuth State Storage And Anti-Replay Foundation

Status: Accepted

## Context

ADR 0046 defined the v1.0 OAuth boundary and callback contract. That contract made OAuth state validation a required dependency before any future callback route can be implemented.

v1.0 Batch 2 establishes the internal OAuth state storage and anti-replay foundation. This foundation is needed so future callback handling can validate state existence, provider ownership, freshness, and single-use semantics before any token exchange or credential storage is considered.

## Decision

v1.0 Batch 2 adds an internal-only OAuth state foundation. It may add a backend service, a digest-only database table, and backend tests. It does not implement real OAuth runtime behavior.

Batch 2 is a prerequisite for a future callback route, but it does not add that route. Future callback route work must remain blocked until a later batch explicitly scopes and tests the route.

OAuth state values must not be raw persisted. The database may store only a digest / hash and safe metadata.

The `provider_oauth_states` table stores:

- `oauth_state_id`
- `provider_id`
- `source_type`
- `state_digest`
- `state_status`
- `purpose`
- `created_at`
- `expires_at`
- `consumed_at`
- `updated_at`
- `safe_status_message`
- `last_status_change_reason`

The state lifecycle supports `pending`, `consumed`, `expired`, and `revoked` or equivalent states. A pending state may be consumed once. A repeated consume attempt must fail as replayed or invalid. Expired state must be rejected and safely marked expired. Missing or malformed state must fail safely. Unsupported providers must fail safely.

`fake_local`, `douyin_sandbox`, and `douyin_real` must remain strictly separated. `douyin_real` must not fallback to `douyin_sandbox`.

## Security Boundary

Batch 2 must not persist:

- Raw OAuth state.
- Authorization code.
- Access token.
- Refresh token.
- Token value.
- Secret.
- API key.
- Credential material.
- Cookie or session values.
- Raw request or raw response payloads.

Raw state may exist only in the internal creation result for a future OAuth handoff. Batch 2 does not expose that value through an API or frontend UI.

Logs, audit records, error responses, tests, fixtures, and docs examples must use only redacted or obvious fake values.

## Non-Goals

This batch does not:

- Implement real OAuth.
- Create an OAuth authorization URL.
- Add an OAuth start route.
- Add an OAuth callback route.
- Add token exchange.
- Store token material.
- Store credential material.
- Store authorization codes.
- Add frontend OAuth UI.
- Call Douyin APIs or external services.
- Upload, publish, schedule publishing, or fetch real metrics.
- Enable `douyin_real`.
- Modify the v0.9.0 tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Claim production, commercial, or SaaS readiness.

## Dependency Gates

- Future callback route implementation still requires a separate batch after this foundation.
- Token exchange must wait for Batch 3.
- Credential reference and encrypted storage must wait for Batch 4.
- Real provider runtime enablement must wait for Batch 5 feature flag / kill switch controls.

## Consequences

The project gains a reviewable internal state foundation for future real OAuth work while still preventing real OAuth runtime behavior.

Future callback implementation can rely on a digest-only state table and single-use consume semantics, but it must still preserve redaction, provider isolation, feature flag / kill switch controls, and user authorization boundaries.
