# ADR 0051: v1.0 User-Confirmed Publish Intent Workflow

## Status

Accepted for v1.0 Batch 6 local workflow foundation.

## Context

v1.0 is the Douyin Integration User Test Release target, but real Douyin runtime behavior remains unavailable. Batch 2 added OAuth state / anti-replay foundation, Batch 3 added a fake-gated token exchange boundary, Batch 4 added credential reference / storage boundary design, and Batch 5 added real provider feature flag / kill switch controls.

Batch 6 establishes the user-confirmed publish intent workflow that future real publishing must pass through before any provider publish adapter is considered. This is a local intent and preflight layer, not a real publish implementation.

## Decision

Batch 6 adds a local-only publish intent workflow. A publish intent may be created only after an explicit user confirmation payload, and it must be linked to an approved review item in the same project. The workflow must perform publish preflight validation, including review readiness, local media artifact readiness, safe metadata validation, provider validation, real provider control checks, and duplicate active intent prevention.

The existing `publish_intents` schema is reused and extended with metadata-only fields for source separation, confirmation status, safe status messages, and status-change reasons. It must not store token, secret, credential, authorization code, raw OAuth state, API key, cookie, session, bearer, raw request, raw response, provider response, upload response, or publish response material.

`douyin_real` remains blocked by Batch 5 controls. `douyin_real` must not fallback to `douyin_sandbox`. `fake_local`, `douyin_sandbox`, and `douyin_real` remain strictly separated.

## Explicit Non-Goals

Batch 6 does not:

- Execute real publishing.
- Upload video.
- Schedule publishing.
- Call Douyin API or any business external service.
- Add a real publish adapter.
- Add OAuth start routes.
- Add OAuth callback routes.
- Create OAuth URLs.
- Implement real OAuth.
- Implement real token exchange.
- Save token, secret, credential, authorization code, raw OAuth state, cookie, session, API key, bearer, raw request, or raw response material.
- Store provider response, upload response, or publish response payloads.
- Add frontend OAuth UI.
- Fetch real metrics.
- Enable `douyin_real`.
- Modify the `v0.9.0` tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Declare production, commercial, or SaaS readiness.

## Safe Workflow Rules

The workflow may return safe metadata such as publish intent id, review item id, provider id, source type, local status, confirmation status, timestamps, safe status message, and status-change reason.

The workflow must reject missing or malformed confirmation, missing or non-approved review items, missing local media artifacts, invalid metadata, duplicate active publish intents, unsupported providers, disabled real providers, and sandbox fallback attempts.

All logs, audit records, responses, tests, fixtures, and documentation examples must stay redacted. Tests may use obvious fake values only for negative assertions.

## Dependency Gates

- Batch 5 real provider controls must continue to block `douyin_real`.
- Batch 7 is required before any guarded real publish adapter implementation.
- Publish status reconciliation must wait for a later accepted batch.
- Real metrics read must wait for a later accepted batch.
- Real OAuth, token exchange, token storage, and credential storage remain governed by the earlier Batch 1-5 boundaries and future accepted implementation work.

## Consequences

Future real publishing must use this local intent layer or an equivalent human-confirmed workflow before invoking any provider adapter. Review approval alone remains insufficient for publishing. Creating a local publish intent remains distinct from upload, real publish, scheduled publish, and metrics read.
