# ADR 0046: v1.0 OAuth Boundary And Callback Contract

Status: Accepted

## Context

v1.0 Batch 0 established that the Douyin Integration User Test Release is a small-scope user test target, not a production commercial release. Batch 0 did not implement real OAuth, callback routes, OAuth state storage, token exchange, credential storage, real provider behavior, publishing, metrics reads, database changes, frontend OAuth UI, or external service calls.

v1.0 Batch 1 defines the boundary and contract for a future real Douyin OAuth authorization start and callback flow. The contract is needed before any runtime route exists so later batches can implement state persistence, anti-replay checks, token exchange, credential storage, feature flag controls, audit, and redaction in the correct order.

## Decision

v1.0 Batch 1 is documentation-only. It defines the OAuth boundary and callback contract for future Douyin real-provider work, but it does not implement real OAuth runtime behavior.

The callback contract is a future implementation constraint only. It does not mean the repository currently has an OAuth start endpoint, OAuth callback route, OAuth error callback route, callback persistence, state storage, token exchange, token storage, credential storage, frontend OAuth UI, real Douyin provider enablement, or any external service call.

Future OAuth authorization start and callback behavior must be designed around these categories:

- OAuth authorization start.
- Success callback with an authorization code and state.
- Error callback from the provider.
- Unsupported provider callback.
- Cancelled or denied authorization.
- Malformed callback.
- Replayed callback.
- Expired callback.

The future callback route must not be implemented until Batch 2 provides OAuth state storage and anti-replay foundation. A callback without state validation is not acceptable for real Douyin integration.

Real token exchange must wait for Batch 3. Credential storage and any encrypted credential material handling must wait for Batch 4. Runtime feature flag and kill switch enablement must wait for Batch 5.

## Contract Boundary

Future OAuth authorization start must only create a provider-specific authorization handoff after feature flag, provider support, redirect URI policy, permission scope policy, and state creation requirements are satisfied. Batch 1 does not create this handoff and does not create a real OAuth URL.

Future callback handling must classify callback inputs without exposing sensitive values:

- A success callback with a code and valid state can only proceed to `pending_token_exchange`.
- A user denied or cancelled callback must map to `authorization_cancelled`.
- A provider error callback must map to `authorization_failed`.
- A malformed callback, missing state, replayed state, expired state, or malformed state must map to `invalid_callback`.
- An unsupported provider must map to `unsupported_provider`.
- A disabled real OAuth entry must map to `oauth_temporarily_disabled`.

The future callback contract must remain source-separated:

- `fake_local` is local fake/demo/test workflow only and does not need OAuth.
- `douyin_sandbox` is sandbox/mock-only and must not be represented as real Douyin OAuth.
- `douyin_real` is the future real provider and must not fallback to `douyin_sandbox`.
- Unknown, unsupported, or disabled providers must fail explicitly.

## Redaction And Audit Boundary

All future logs, audit records, error responses, fixtures, screenshots, and examples must be redacted. They must not contain real access tokens, refresh tokens, token values, secrets, API keys, credentials, authorization codes, OAuth state values, cookies, sessions, bearer values, raw provider request payloads, or raw provider response payloads.

User-facing responses must not display code, state, token, secret, credential, cookie, session, or raw provider payload values. Logs and audit metadata may contain safe category, provider id, status, timestamp, request correlation id, and redacted error class only.

Tests may use obvious fake values only, such as `fake_code_for_contract_test` and `fake_state_for_contract_test`, and must never use values copied from real provider callbacks.

## Feature Flag And Kill Switch Boundary

Future real OAuth entry points must default to disabled until the relevant later batch accepts runtime enablement. The kill switch must be able to block authorization start and callback continuation before any token exchange or credential storage can occur.

When real OAuth is disabled, the safe outcome is `oauth_temporarily_disabled`. The disabled path must not fallback to `douyin_sandbox`, must not create a token exchange attempt, and must not persist callback-sensitive material.

## Batch 1 Non-Goals

This batch does not:

- Implement real OAuth.
- Create an OAuth URL.
- Add an OAuth callback route.
- Add OAuth state storage.
- Add token exchange.
- Store tokens.
- Store secrets.
- Store credentials.
- Store authorization codes.
- Store OAuth state values.
- Read real environment secrets.
- Call Douyin APIs or external services.
- Add backend APIs.
- Add frontend OAuth UI.
- Modify database tables or migrations.
- Upload, publish, schedule publishing, or fetch real metrics.
- Modify the v0.9.0 tag.
- Declare v1.0, v1.5, or v2.0 complete.
- Claim production, commercial, or SaaS readiness.

## Consequences

Later v1.0 batches have a safe implementation order:

1. Batch 2 must provide OAuth state storage and anti-replay foundation before any callback route can be implemented.
2. Batch 3 must define and gate token exchange before any code can be exchanged.
3. Batch 4 must define credential reference and encrypted storage before any token or credential material can be stored.
4. Batch 5 must define runtime feature flag and kill switch controls before real OAuth can be enabled for testers.

This keeps the v1.0 user test path reviewable, testable, redacted, and clearly separated from production, v1.5 Minimum Production Release, and v2.0 Multi-Tenant SaaS Commercial Release.
