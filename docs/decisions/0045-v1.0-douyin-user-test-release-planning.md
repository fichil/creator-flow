# ADR 0045: v1.0 Douyin User Test Release Planning

Status: Accepted

## Context

v0.9.0 has been released as Douyin Provider POC / Sandbox Integration from release commit `0f5263452b65077f2c70c82e506944dd46e60e96`. That release provides sandbox-only deterministic provider descriptors, mock connection, metrics preview, publish dry-run, registry / factory routing, frontend sandbox POC review surface, validation script, and release review documentation.

v0.9.0 is not real Douyin integration. It does not implement real OAuth, create OAuth URLs, add OAuth callback routes, add OAuth state storage, exchange or store tokens, store credentials, call Douyin APIs, fetch real metrics, upload, publish, schedule publishing, or call external services.

The next stage is v1.0 Douyin Integration User Test Release planning. v1.0 is intended to validate a real Douyin integration path with a small group of internal / test users, but it is not a production commercial release.

## Decision

v1.0 Batch 0 is docs-only / planning-only. It plans the Douyin Integration User Test Release and does not implement runtime behavior.

v1.0 targets a small-scope user test of the real Douyin integration path. The planned user test should verify whether explicit user authorization, real account connection status, user-confirmed publish workflow, publish status reconciliation, and limited metrics read are feasible under Douyin Open Platform permissions and review constraints.

v1.0 is not a production commercial release. v1.0 is not v1.5 Minimum Production Release. v1.0 is not v2.0 Multi-Tenant SaaS Commercial Release.

## Required Preconditions For Later Real Implementation

Future v1.0 implementation batches must be gated by:

- Douyin Open Platform app readiness.
- App review / approval.
- OAuth permission scope confirmation.
- Callback URL confirmation.
- User authorization consent design.
- OAuth state anti-replay design.
- Token lifecycle policy.
- Credential reference and encrypted storage design.
- Platform rate limit and error policy.
- Audit log design with redaction.
- Feature flag and kill switch design.
- Clear source separation between `fake_local`, `douyin_sandbox`, and `douyin_real`.

If platform permissions, app review, callback URL approval, OAuth scopes, or explicit user authorization are unavailable, the corresponding real capability must remain blocked or deferred.

## Batch 0 Boundary

This batch does not:

- Implement real OAuth.
- Create OAuth URLs.
- Add OAuth callback routes.
- Add OAuth state storage.
- Add token exchange.
- Add token storage.
- Add credential storage.
- Read real environment secrets.
- Call Douyin APIs or external services.
- Upload, publish, schedule publishing, or fetch real metrics.
- Modify database tables or migrations.
- Add backend APIs.
- Add frontend OAuth UI.
- Change the v0.9.0, v0.8.0, or v0.7.0 tags.
- Declare v1.0, v1.5, or v2.0 complete.
- Claim production, commercial, or SaaS readiness.

## Relationship To v1.5 And v2.0

v1.0 is only a user test release target. It may inform later hardening work, but it does not establish production operations, customer acceptance, support process, SLA, backup / restore, monitoring, incident response, legal / commercial terms, or controlled direct-customer commercial readiness.

v1.5 remains the future Minimum Production Release target for controlled direct-customer commercial use after separate readiness criteria are satisfied.

v2.0 remains the future Multi-Tenant SaaS Commercial Release target for customer-of-customer SaaS commercialization after tenant isolation, RBAC, billing, audit, support, SLA, compliance, operations, and SaaS readiness criteria are satisfied.

## Consequences

The project can start v1.0 planning from the released v0.9.0 sandbox POC baseline without prematurely adding real platform capability.

Future v1.0 Batch 1-N work must stay narrowly scoped, separately reviewable, testable, and security-scanned. Each real platform capability must have an accepted ADR before implementation and must preserve human-in-the-loop publishing.

## Follow-up Work

- v1.0 Batch 1: OAuth boundary design and callback contract.
- v1.0 Batch 2: OAuth state storage and anti-replay foundation.
- v1.0 Batch 3: Token exchange boundary with fake-gated integration.
- v1.0 Batch 4: Credential reference and encrypted storage design.
- v1.0 Batch 5: Real provider feature flag / kill switch.
- v1.0 Batch 6: User-confirmed publish intent workflow.
- v1.0 Batch 7: Real publish adapter guarded implementation.
- v1.0 Batch 8: Publish status reconciliation.
- v1.0 Batch 9: Limited real metrics read.
- v1.0 Batch 10: User test readiness / release candidate.
