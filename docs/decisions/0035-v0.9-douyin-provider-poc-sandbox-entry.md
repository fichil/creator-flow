# ADR 0035: v0.9 Douyin Provider POC and Sandbox Integration Entry

## Status

Accepted

## Context

v0.8.0 has been released as Provider & Credential Security Foundation. It established the metadata-only / read-only Provider Registry, Connection State, Credential Reference, Security Audit, OAuth Boundary, Token Lifecycle Boundary, Readiness Summary, source separation, secret redaction, frontend read-only panels, RC checklist, and release decision.

v0.8.0 is not real Douyin integration. It does not implement OAuth, add OAuth callback routes, store tokens, fetch real metrics, upload, publish, schedule, or call external services.

v0.9 is the next stage for Douyin Provider POC / Sandbox Integration. It must begin with planning, ADR, and checklist work before any implementation. The first implementation work must prioritize sandbox/mock boundaries and dry-runs rather than direct real Douyin integration.

## Decision

v0.9 may start only after the v0.8.0 release.

v0.9 Batch 0 is planning / ADR / checklist only.

v0.9 future work must start with sandbox/mock integration before real Douyin.

v0.9 must preserve `fake_local` / `douyin_sandbox` / `douyin_real` separation.

v0.9 must not store real tokens until encrypted credential storage is separately designed and accepted.

v0.9 must not call real Douyin API until provider adapter and OAuth boundaries are separately approved.

Any real platform action must be introduced through a separate ADR, separate branch, separate tests, and separate security scan.

## v0.9 Scope

v0.9 Batch 0 may add:

- Douyin Provider POC / Sandbox Integration planning documentation.
- A v0.9 entry ADR.
- A v0.9 POC readiness checklist.
- README, roadmap, architecture, product spec, and development documentation updates.
- Boundaries for future Douyin provider adapter skeleton planning.
- Boundaries for future sandbox/mock callback smoke tests.
- Boundaries for future provider status transition dry-runs.
- Boundaries for future metrics read POC planning.
- Boundaries for future read-only mock/sandbox readiness display.

v0.9 Batch 0 does not implement those future capabilities.

## Sandbox / Mock Boundary

Sandbox/mock integration is the required first implementation direction after Batch 0. It may only validate provider contract shape, callback planning, redacted error handling, source labeling, and provider status transition dry-run behavior after a separate implementation ADR.

Sandbox/mock callback behavior must not be represented as real OAuth. Sandbox/mock data must not be displayed as real Douyin data. Sandbox/mock readiness must remain clearly labeled and must not imply user-facing real platform availability.

Batch 0 adds no callback route, no callback parser, no OAuth state storage, no authorization code storage, no token exchange, and no platform request.

## Real Douyin Boundary

`douyin_real` remains future real provider placeholder behavior. It must not be treated as available, connected, authorized, token stored, metrics ready, publish ready, scheduled, or production ready.

Real Douyin API calls require a separate provider adapter ADR, OAuth boundary approval, tests, security scan, and platform permission review before implementation. Real upload, publish, scheduling, or automated publishing are outside this entry decision and remain prohibited.

## OAuth Boundary

v0.9 Batch 0 does not implement OAuth.

No OAuth start route, authorize route, callback route, OAuth state storage, state validator, authorization code handler, token exchange, token refresh, token revoke, disconnect, or provider authorization URL generation is added in Batch 0.

Future OAuth work must separately define state storage, CSRF protection, redirect URI policy, callback error handling, token exchange, credential storage, logging redaction, and test evidence before any real callback or token exchange is implemented.

## Credential and Token Boundary

v0.9 Batch 0 stores no token, secret, API key, credential, authorization code, OAuth state, credential material, encrypted credential, private key, raw request, raw response, raw payload, token expiry value, token refresh response, token revoke response, or provider token response.

Real access tokens and refresh tokens must not be stored until encrypted credential storage is separately designed, accepted, implemented, tested, and scanned. Metadata-only credential references from v0.8 must not be reinterpreted as real Credential storage.

## Security Audit Boundary

Security audit behavior remains metadata-only unless a later ADR approves implementation changes. Any future sandbox/mock or real provider event must pass redaction before storage or display and must preserve provider/source separation.

Logs, errors, fixtures, examples, screenshots, and documentation must not contain real secrets, real OAuth callback data, real provider responses, real Douyin credentials, or real platform payloads.

## Source Separation

`fake_local` remains local fake/demo/test workflow only.

`douyin_sandbox` remains sandbox/mock or placeholder only.

`douyin_real` remains future real provider only.

The three sources must not impersonate each other. `fake_local` readiness is not real Douyin readiness. `douyin_sandbox` readiness is not real OAuth, real token lifecycle, real metrics, real upload, real publish, or real scheduling readiness. `douyin_real` must remain blocked until real provider adapter, OAuth, credential storage, token lifecycle, permission, and testing boundaries are separately accepted.

- Do not add OAuth state storage.
- Do not add token exchange.
- Do not store token.
- Do not store secret.
- Do not store API key.
- Do not store credential.
- Do not store authorization code.
- Do not store OAuth state.
- Do not store access token.
- Do not store refresh token.
- Do not store token value.
- Do not store credential material.
- Do not store encrypted credential.
- Do not store private key.
- Do not store raw request.
- Do not store raw response.
- Do not store raw payload.
- Do not fetch real metrics.
- Do not upload.
- Do not publish.
- Do not schedule publishing.
- Do not auto-publish.
- Do not call external services.
- Do not add business code.
- Do not add backend APIs.
- Do not modify database tables.

## Consequences

v0.9 starts from a reviewable entry point rather than an implementation jump.

Future contributors can plan Douyin Provider POC work while preserving v0.8 security boundaries, source separation, and human-in-the-loop publishing principles.

The first future implementation batches must remain sandbox/mock-first and must avoid real Douyin calls, real OAuth, real token storage, real metrics fetching, upload, publish, scheduling, or external service calls unless explicitly approved by follow-up ADRs.

## Required Follow-up ADRs

- Douyin Provider Adapter Skeleton ADR.
- Sandbox / Mock OAuth Callback Smoke Test ADR.
- Credential Storage / Encryption ADR.
- Token Exchange and Refresh ADR.
- Real Metrics Read POC ADR.
- Disconnect / Revoke Lifecycle ADR.
