# ADR 0033: v0.8 Provider Security Foundation RC Audit

## Status

Accepted

## Context

v0.7.0 已完成 fake/local metrics review summary workflow，并且当前稳定版本仍是 `v0.7.0 - local fake/manual metrics review summary workflow`。

v0.8 Batch 1 through Batch 15 have established the Provider & Credential Security Foundation as a sequence of documentation, backend metadata, read-only API, and frontend read-only UI boundaries:

- Batch 1: Provider / Credential / OAuth / Secret / token lifecycle documentation boundary.
- Batch 2 and Batch 3: Provider Registry backend and frontend read-only foundation.
- Batch 4 and Batch 5: Provider Connection State backend and frontend read-only foundation.
- Batch 6 and Batch 7: Provider Credential Reference, Secret Redaction, and frontend read-only foundation.
- Batch 8 and Batch 9: Provider Security Audit Event / Redacted Audit Log backend and frontend read-only foundation.
- Batch 10 and Batch 11: Provider OAuth State & Callback Boundary backend and frontend read-only foundation.
- Batch 12 and Batch 13: Provider Token Lifecycle Boundary backend and frontend read-only foundation.
- Batch 14 and Batch 15: Provider Integration Readiness Summary backend and frontend read-only foundation.

Before v0.9 Douyin Provider POC / Sandbox Integration begins, the project needs a release-candidate audit artifact that verifies the Batch 1-15 foundation is internally consistent, still read-only / metadata-only, and still does not imply real Douyin, real OAuth, real token lifecycle, real Credential storage, real metrics fetching, upload, publish, scheduling, v0.9 POC completion, or production readiness.

## Decision

Batch 16 adds a docs-only RC audit / closure checklist for v0.8 Provider & Credential Security Foundation and an ADR recording the closure boundary.

The RC audit covers:

- Batch 1-15 branch and commit inventory.
- Batch scope classification as docs-only, backend-only, frontend-only, or frontend + docs.
- Read-only API inventory.
- Frontend read-only panel inventory.
- Metadata-only DB boundary.
- Response schema sensitive-field boundary.
- Documentation consistency across README, roadmap, architecture, product spec, development docs, checklists, and ADRs.
- Explicit non-goals and security scan expectations.

Batch 16 does not add business code, backend API, database tables, frontend UI, real Provider, real Provider adapter, OAuth implementation, OAuth callback route, OAuth state storage, token exchange, token lifecycle workflow, real Douyin integration, external service calls, metrics fetching, upload, publish, scheduling, `v0.8.0` tag, v0.8 release declaration, or v0.9 POC development.

## Consequences

The project has a single RC checklist that reviewers can use to verify v0.8 Provider & Credential Security Foundation before any v0.9 work begins.

The checklist makes it explicit that v0.8 is a security and metadata foundation, not real platform integration. It also keeps the current stable release anchored to v0.7.0 until a separate release process intentionally creates and announces a later version.

Future v0.9 work can reference the RC checklist, but must still introduce separate ADRs, tests, security review, and scope boundaries before adding a provider adapter skeleton, sandbox/mock callback smoke test, token lifecycle dry-run, or real metrics POC.

## RC Checklist Boundary

The RC checklist is an audit artifact. It can describe completed batches, expected validation gates, and security-scan criteria, but it must not be treated as a release tag, production readiness certification, or proof that real provider integration is available.

The checklist must explicitly cover:

- Whether Batch 1-15 are complete.
- Whether each batch has branch, commit, scope, test evidence category, and boundary confirmation.
- Whether backend-only, frontend-only, frontend + docs, and docs-only scopes are consistent.
- Whether read-only APIs remain read-only and lack write operations.
- Whether frontend panels lack operation buttons, token viewers, secret inputs, and credential management UI.
- Whether DB tables remain metadata-only and exclude sensitive-value field names.
- Whether response schemas exclude fields that can carry sensitive values.
- Whether docs consistently state that v0.8 is not real Douyin integration.
- Whether README still states current stable version is v0.7.0.
- Whether roadmap still states v0.9 is the Douyin Provider POC / Sandbox Integration stage.
- Whether security scan confirms no real sensitive value or runtime artifact entered Git.

## Documentation Consistency Boundary

README, README.en, roadmap, architecture, product spec, development docs, checklist, and ADRs must keep the same core story:

- v0.8 is Provider & Credential Security Foundation.
- v0.8 is not a release announcement.
- v0.8 is not real Douyin integration.
- v0.8 is not OAuth implementation.
- v0.8 is not Credential storage.
- v0.8 is not token storage.
- v0.8 is not real metrics fetching.
- v0.8 is not upload, publish, scheduling, or automatic publishing.
- v0.9 is where Douyin Provider POC / Sandbox Integration begins.
- v1.0 is the Douyin Integration User Test Release direction.

## API Boundary

The RC audit recognizes the v0.8 read-only metadata API set:

- `GET /api/providers`
- `GET /api/providers/{provider_id}`
- `GET /api/provider-connections`
- `GET /api/provider-connections/{provider_id}`
- `GET /api/provider-credential-references`
- `GET /api/provider-credential-references/{provider_id}`
- `GET /api/provider-security-audit-events`
- `GET /api/provider-security-audit-events/{audit_event_id}`
- `GET /api/provider-oauth-boundaries`
- `GET /api/provider-oauth-boundaries/{provider_id}`
- `GET /api/provider-token-lifecycle-boundaries`
- `GET /api/provider-token-lifecycle-boundaries/{provider_id}`
- `GET /api/provider-readiness-summaries`
- `GET /api/provider-readiness-summaries/{provider_id}`

The audit must not add POST, PUT, PATCH, DELETE, connect, authorize, callback, token exchange, refresh, revoke, disconnect, credential write, token read, secret read, upload, publish, schedule, readiness approval, readiness override, certification, or v0.9 POC start routes.

## Frontend Boundary

The RC audit recognizes the v0.8 read-only frontend panel set:

- Provider Registry.
- Provider Connection State.
- Provider Credential Reference.
- Provider Security Audit.
- Provider OAuth Boundary.
- Provider Token Lifecycle Boundary.
- Provider Integration Readiness Summary.

The audit must not add frontend UI. Existing panels must remain read-only metadata displays and must not provide action buttons, secret inputs, token viewers, credential management UI, raw request / response / payload viewers, readiness override / approval / certification UI, or v0.9 POC start UI.

## Data Boundary

Metadata tables and computed readiness summaries may describe status and policy. They must not store access token, refresh token, token value, API key, secret, authorization code, OAuth client secret, OAuth state value, credential material, encrypted credential, private key, raw request, raw response, raw payload, token expiry value, token refresh response, token revoke response, or provider token response.

`fake_local`, `douyin_sandbox`, and `douyin_real` must remain explicit and separate. `fake_local` local readiness must not become real provider readiness. `douyin_sandbox` and `douyin_real` placeholder metadata must not become connected, authorized, token stored, token refreshed, real synced, published, v0.9 POC complete, or production ready.

## Testing Requirements

Batch 16 must run the full local gate:

- Backend tests with `.\.venv\Scripts\python.exe -m pytest`.
- Frontend tests with `npm.cmd run test -- --run`.
- Frontend build with `npm.cmd run build`.
- `git diff --check`.
- `git status --short`.
- Security scan for sensitive values and runtime artifacts.

If any gate fails, the RC audit is not complete until it is fixed and rerun. Security scan matches in documentation boundary language, test deny lists, fake test values, ignore rules, or dependency metadata are acceptable only after confirming they are not real secrets or runtime files.
