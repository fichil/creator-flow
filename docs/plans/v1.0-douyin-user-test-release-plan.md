# v1.0 Douyin User Test Release Plan

## v1.0 Goal

v1.0 targets a Real Douyin Integration User Test Release for a small group of internal / test users.

The goal is to validate whether the real Douyin integration path is feasible under explicit user authorization, Douyin Open Platform permissions, app review constraints, and human-in-the-loop publishing rules.

The user test should verify:

- Real authorization path feasibility.
- Real publish path feasibility after explicit user confirmation.
- Real publish status query feasibility.
- Limited metrics read feasibility.
- Strict separation between real provider behavior, sandbox provider behavior, and local fake workflow.

## v1.0 Non-Goals

- No production commercial commitment.
- No SLA commitment.
- No multi-tenant SaaS commitment.
- No customer-of-customer commercial use.
- No batch automatic publishing.
- No bypassing platform review, permissions, rate limits, or user authorization.
- No claim that v1.0 is v1.5 Minimum Production Release.
- No claim that v1.0 is v2.0 Multi-Tenant SaaS Commercial Release.

## Key Preconditions

- Douyin Open Platform app readiness.
- App review / approval.
- OAuth permission scope confirmation.
- Callback URL confirmation.
- Token lifecycle policy.
- Credential encryption / storage design.
- User authorization consent design.
- Platform rate limit / error policy.
- Audit log design.
- Kill switch / feature flag design.

Capabilities that depend on Douyin platform permissions, app review, callback URL approval, OAuth scopes, or explicit user authorization must remain blocked until those prerequisites are satisfied.

## Suggested v1.0 Batch Route

1. v1.0 Batch 1: OAuth boundary design and callback contract. Batch 1 defines ADR 0046, the future callback contract, and a contract test matrix only; it does not implement real OAuth runtime behavior.
2. v1.0 Batch 2: OAuth state storage and anti-replay foundation. Batch 2 defines ADR 0047, an internal-only state storage contract, a digest-only schema, an internal state service, and anti-replay tests; it does not add OAuth routes or real OAuth runtime behavior.
3. v1.0 Batch 3: Token exchange boundary with fake-gated integration. Batch 3 defines ADR 0048, an internal-only token exchange boundary contract, a fake-gated simulator, and backend tests; it does not add real token exchange, token storage, credential storage, OAuth routes, OAuth URLs, frontend OAuth UI, real provider calls, publishing, or metrics.
4. v1.0 Batch 4: Credential reference and encrypted storage design. Batch 4 defines ADR 0049, an internal-only credential storage boundary contract, metadata-only reference behavior, raw material rejection, encryption-provider-unavailable guard, and backend tests; it does not add real token storage, real credential storage, encrypted payload storage, OAuth routes, OAuth URLs, frontend OAuth UI, real provider calls, publishing, or metrics.
5. v1.0 Batch 5: Real provider feature flag / kill switch. Batch 5 defines ADR 0050, an internal-only real provider controls contract, default real provider blocking, kill switch precedence, missing / malformed flag safe blocking, platform preconditions blocking, sandbox fallback forbidden behavior, and backend tests; it does not enable real provider runtime, OAuth routes, OAuth URLs, real token exchange, real token storage, real credential storage, frontend OAuth UI, real provider calls, publishing, or metrics.
6. v1.0 Batch 6: User-confirmed publish intent workflow. Batch 6 defines ADR 0051, a local-only publish intent contract, metadata-only publish intent schema extension, explicit confirmation requirement, review-state linkage, preflight validation, duplicate active intent prevention, real provider disabled handling, sandbox fallback forbidden behavior, backend tests, and frontend UI/tests; it does not upload, real publish, schedule publishing, call Douyin, create OAuth URLs/routes, perform token exchange, store credentials, or read real metrics.
7. v1.0 Batch 7: Real publish adapter guarded implementation. Current Batch 7 defines ADR 0052, a guarded real publish adapter contract, metadata-only publish attempt schema, confirmed publish intent dependency, review / media / metadata / credential reference preflight, duplicate attempt prevention, real provider disabled handling, kill switch / platform precondition blocking, sandbox fallback forbidden behavior, backend tests, and frontend UI/tests; it does not upload, real publish, schedule publishing, call Douyin, create OAuth URLs/routes, perform token exchange, store credentials, store provider / upload / publish responses, or read real metrics.
8. v1.0 Batch 8: Publish status reconciliation.
9. v1.0 Batch 9: Limited real metrics read.
10. v1.0 Batch 10: User test readiness / release candidate.

Each batch should have a separate ADR, tests, documentation updates, and security scan. Batch 0 does not implement any runtime capability. Batch 1 remains docs-only / contract-only. Batch 2 adds internal-only state foundation capability. Batch 3 adds internal-only token exchange boundary capability with fake-gated simulation. Batch 4 adds internal-only credential reference / storage boundary capability. Batch 5 adds internal-only real provider controls. Batch 6 adds a local-only publish intent workflow. Batch 7 adds a guarded publish attempt foundation, and still does not implement OAuth URLs, OAuth start routes, callback routes, real token exchange, real token storage, real credential storage, frontend OAuth UI, real provider runtime behavior, real publishing, upload, scheduled publishing, metrics, or external service calls.

## Batch 1 OAuth Boundary / Callback Contract

Batch 1 defines the future OAuth boundary and callback contract for the Douyin Integration User Test Release.

Batch 1 covers future contract categories only:

- OAuth authorization start.
- Success callback with `code` and `state`.
- User denied / cancelled authorization.
- Provider error callback.
- Malformed callback.
- Missing state.
- Replayed state.
- Expired state.
- Unsupported provider.

Expected safe response categories are `pending_token_exchange`, `authorization_cancelled`, `authorization_failed`, `invalid_callback`, `unsupported_provider`, and `oauth_temporarily_disabled`.

Batch 1 dependency gates after Batch 2:

- Batch 2 supplies the state persistence / anti-replay foundation, but callback route implementation still requires a later accepted batch.
- Batch 3 must exist before token exchange.
- Batch 4 must exist before credential storage.
- Batch 5 must exist before runtime feature flag / kill switch enablement for real OAuth.

Batch 1 redaction and isolation rules:

- User-facing responses, logs, audit records, screenshots, fixtures, and examples must not show token, secret, credential, authorization code, OAuth state, cookie, session, raw request, or raw response values.
- Tests may use obvious fake values only.
- `fake_local`, `douyin_sandbox`, and `douyin_real` must remain strictly separated.
- `douyin_real` must not fallback to `douyin_sandbox`.

Batch 1 non-goals:

- No real OAuth.
- No OAuth URL.
- No OAuth callback route.
- No OAuth state storage.
- No token exchange.
- No token storage.
- No credential storage.
- No external service call.
- No DB change.
- No frontend OAuth UI.

## Batch 2 OAuth State Storage / Anti-Replay Foundation

Batch 2 establishes the internal-only OAuth state storage and anti-replay foundation required before any future callback route can be implemented.

Batch 2 covers foundation categories only:

- Internal state creation for future handoff.
- Digest-only state persistence.
- Known provider validation.
- One-time state consume.
- Replay rejection.
- Expiry rejection and safe expired marking.
- Missing / malformed state rejection.
- Provider mismatch rejection.
- Unsupported provider rejection.
- Strict `fake_local`, `douyin_sandbox`, and `douyin_real` source separation.

Batch 2 storage rules:

- Raw OAuth state is never persisted.
- The `provider_oauth_states` table stores only digest and safe metadata.
- Metadata includes provider id, source type, state status, purpose, creation time, expiry time, optional consumed time, updated time, safe status message, and last status change reason.
- `douyin_real` must not fallback to `douyin_sandbox`.
- Safe messages, logs, audit records, fixtures, and examples must not contain raw state, authorization code, token, secret, credential, cookie, session, raw request, or raw response values.

Batch 2 dependency gates:

- Future callback routes may use the Batch 2 state foundation but still require a later accepted route batch.
- Batch 3 must exist before token exchange.
- Batch 4 must exist before credential storage.
- Batch 5 must exist before runtime feature flag / kill switch enablement for real OAuth.

Batch 2 non-goals:

- No real OAuth.
- No OAuth URL.
- No OAuth start route.
- No OAuth callback route.
- No token exchange.
- No token storage.
- No credential storage.
- No frontend OAuth UI.
- No external service call.
- No publish, metrics, upload, or scheduling.

## Batch 3 Token Exchange Boundary / Fake-Gated Integration

Batch 3 establishes the internal-only token exchange boundary with fake-gated integration required before any future real token exchange can be implemented.

Batch 3 covers foundation categories only:

- Known provider validation.
- Strict source type separation for `fake_local`, `douyin_sandbox`, and `douyin_real`.
- Batch 2 state validation dependency before exchange simulation.
- Missing / malformed state rejection.
- Replayed state rejection.
- Expired state rejection.
- Provider mismatch rejection.
- Unsupported provider rejection.
- Missing / malformed authorization code rejection.
- Default `douyin_real` disabled / blocked result.
- Sandbox fallback forbidden behavior.
- Metadata-only / redacted exchange simulation for allowed fake / sandbox-safe inputs.

Batch 3 result rules:

- Fake-gated exchange may return only safe metadata such as provider id, source type, safe status category, safe status message, whether a simulated token response was received, and whether future credential storage is required.
- Service output must not contain authorization code, raw OAuth state, access token, refresh token, token, secret, credential, cookie, session, raw request, or raw response values.
- No token material is persisted, logged, audited, returned, or included in fixtures.
- `douyin_real` remains disabled / blocked and must not fallback to `douyin_sandbox`.
- Batch 3 does not add any DB schema.

Batch 3 dependency gates:

- Batch 2 state foundation must be used before fake-gated exchange simulation.
- Batch 4 must exist before credential reference / encrypted storage.
- Batch 5 must exist before runtime feature flag / kill switch enablement for real provider behavior.
- Any future real token exchange still requires a separate accepted implementation batch, platform readiness, app review, permission confirmation, security review, and redaction tests.

Batch 3 non-goals:

## Batch 4 Credential Reference / Encrypted Storage Design

Batch 4 establishes the internal-only credential reference and encrypted storage boundary required before any future real credential persistence can be implemented.

Batch 4 covers foundation categories only:

- Known provider validation.
- Strict source type separation for `fake_local`, `douyin_sandbox`, and `douyin_real`.
- Batch 3 safe token exchange metadata dependency before reference creation.
- Missing / malformed token exchange metadata rejection.
- Raw access token / refresh token rejection.
- Raw authorization code rejection.
- Raw OAuth state rejection.
- Secret, client secret, credential, cookie, session, bearer, raw request, and raw response rejection.
- Default `douyin_real` disabled / blocked result.
- Sandbox fallback forbidden behavior.
- Encryption-provider-unavailable safe result.
- Metadata-only credential reference creation for allowed fake / sandbox-safe inputs.

Batch 4 storage rules:

- Credential reference rows are metadata-only and may use the existing `provider_credential_references` table.
- Metadata may include provider id, source type, reference kind, reference status, storage status, redaction policy status, safe display name, safe status message, state-change reason, and timestamps.
- Service output and DB metadata must not contain authorization code, raw OAuth state, access token, refresh token, token, secret, credential, client secret, cookie, session, bearer, raw request, or raw response values.
- No encrypted payload table is added in Batch 4.
- No token or credential material is persisted, logged, audited, returned, or included in fixtures.
- `douyin_real` remains disabled / blocked and must not fallback to `douyin_sandbox`.
- If no accepted encryption provider and key-management design exists, Batch 4 must not implement pseudo-encryption. Base64, hashing, XOR, ordinary encoding, and custom algorithms are not acceptable substitutes.

Batch 4 dependency gates:

- Batch 2 state foundation must exist before callback state handling.
- Batch 3 token exchange boundary metadata must be used before credential reference handoff.
- Batch 5 must exist before runtime feature flag / kill switch enablement for real provider behavior.
- Any future real token storage or real credential storage still requires a separate accepted implementation batch, reliable encryption / key-management design, platform readiness, app review, permission confirmation, security review, and redaction tests.

Batch 4 non-goals:

- No real OAuth.
- No OAuth URL.
- No OAuth start route.
- No OAuth callback route.
- No real token exchange.
- No real token storage.
- No real credential storage.
- No encrypted payload storage.
- No frontend OAuth UI.
- No external service call.
- No publish, metrics, upload, or scheduling.

- Missing / malformed / active / revoked / expired / policy-denied kill switch rejection.

- Unsupported capability rejection.

Batch 5 control rules:

- Control decisions must not contain token, secret, credential, authorization code, raw OAuth state, cookie, session, API key, bearer, raw request, or raw response values.

## Risks

- Douyin API permissions may be unavailable.
- OAuth app review may fail or be delayed.
- Token storage may introduce security risk.
- Publish behavior may be irreversible after the platform accepts it.
- Metrics freshness may be delayed, partial, or unstable.
- Rate limits or platform policy may change.
- Users may mistake v1.0 for production availability.
- Sandbox and real provider behavior may be confused if source separation is weak.

## Security Principles

- User authorization must be explicit.
- Tokens must never be exposed to the frontend.
- Business tables must not store secret material directly.
- Credentials must be encrypted or stored behind an accepted secure storage design.
- OAuth state must protect against replay.
- All real publishing must be user-confirmed.
- Real provider feature flags default to disabled until accepted for user testing.
- A kill switch must be able to disable real integration quickly.
- All real platform responses must be redacted before logging, audit display, or user-facing error output.
- All real platform calls must produce audit log coverage with safe metadata only.
- `fake_local`, `douyin_sandbox`, and `douyin_real` must remain strictly separated.

## Acceptance Criteria

- All v0.9 regression checks pass.
- OAuth flow is testable by approved user testers after platform prerequisites are met.
- Token lifecycle behavior is safe, observable, and controllable.
- Publish requires explicit user confirmation.
- Real provider and sandbox provider remain strictly isolated.
- Documentation clearly states that v1.0 is not a production commercial release.
- Security scans show no real credential leakage.
- The real provider can be disabled when errors, policy risk, or credential risk appear.

## Batch 0 Boundary

v1.0 Batch 0 is docs-only / planning-only. It adds planning and ADR coverage for the Douyin Integration User Test Release. It does not add business code, backend APIs, frontend UI, database tables, migrations, real OAuth, OAuth URLs, callback routes, OAuth state storage, token exchange, token storage, credential storage, real provider behavior, external service calls, real metrics reads, uploads, publishes, or scheduled publishing.
