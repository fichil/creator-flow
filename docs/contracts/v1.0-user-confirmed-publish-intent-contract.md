# v1.0 User-Confirmed Publish Intent Contract

This document defines the v1.0 Batch 6 user-confirmed publish intent boundary. It is a local workflow contract, not a Douyin API contract.

Batch 6 does not add real OAuth, OAuth URL creation, OAuth start routes, OAuth callback routes, real token exchange, real token storage, real credential storage, real publish, upload, scheduled publish, metrics read, or external service calls.

## Boundary

The workflow may create a metadata-only local publish intent after explicit user confirmation. The intent must be linked to a review item in the same project, and the review item must be approved before the intent can be created. A local media artifact must be ready before the intent is accepted.

Creating a publish intent does not execute a provider publish call. It does not upload media, schedule publishing, call Douyin, or read metrics.

## Input Categories

| Input category | Expected boundary handling |
| --- | --- |
| Valid review item plus explicit confirmation | Create metadata-only local publish intent. |
| Missing confirmation | Return `confirmation_required`. |
| Malformed confirmation | Return a safe validation failure. |
| Review item missing | Return a safe not-found / not-ready result. |
| Review item not approved or not ready | Return `review_not_ready`. |
| Media artifact missing | Return `media_not_ready`. |
| Caption or metadata invalid | Return `metadata_invalid`. |
| Duplicate active publish intent | Return `duplicate_publish_intent`. |
| Unsupported provider | Return `unsupported_provider`. |
| Real provider disabled | Return `real_provider_disabled`. |
| Sandbox fallback attempt | Return `sandbox_fallback_forbidden`. |

## Expected Safe Result Categories

- `publish_intent_created`
- `publish_intent_blocked`
- `confirmation_required`
- `review_not_ready`
- `media_not_ready`
- `metadata_invalid`
- `duplicate_publish_intent`
- `unsupported_provider`
- `real_provider_disabled`
- `sandbox_fallback_forbidden`

Implementation result names may be equivalent, but user-facing and test-visible messages must remain safe and redacted.

## Redaction Rules

User-facing responses must not display token, access token, refresh token, secret, client secret, credential, authorization code, raw OAuth state, OAuth state value, cookie, session, API key, bearer, raw request, raw response, provider response, upload response, or publish response values.

Logs and audit records must not display sensitive material.

Database metadata must not store raw authorization code, raw OAuth state, token, secret, credential, cookie, session, API key, bearer, raw request, raw response, provider response, upload response, or publish response material.

Tests may use only obvious fake values for negative assertions. Test fixtures must not copy values from real provider callbacks, token responses, cookies, sessions, platform payloads, local secrets, or environment secrets.

## Explicit Non-Goals

Batch 6 does not provide:

- Real OAuth.
- OAuth URL creation.
- OAuth start route.
- OAuth callback route.
- Real token exchange.
- Real token storage.
- Real credential storage.
- Real publish.
- Upload.
- Scheduled publish.
- Metrics read.
- External service calls.
- Frontend OAuth UI.
- Production, commercial, or SaaS readiness.

## Future Dependency Gates

Batch 7 must define and accept a guarded real publish adapter before any real provider publish call exists. Publish status reconciliation, limited real metrics read, audit expansion, and user test release candidate work remain later batches.
