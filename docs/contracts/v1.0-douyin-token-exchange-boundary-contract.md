# v1.0 Douyin Token Exchange Boundary Contract

This document defines the internal-only token exchange boundary for v1.0 Batch 3. It is not a public API contract.

Batch 3 does not add an OAuth start endpoint, OAuth callback endpoint, OAuth URL creation, real token exchange, token storage, credential storage, frontend OAuth UI, external service call, publish workflow, upload workflow, or metrics read. Batch 4 adds a separate internal-only credential storage boundary, and Batch 5 adds a separate internal-only real provider controls boundary, but they still do not add real token storage, real credential storage, encrypted payload storage, OAuth routes, frontend OAuth UI, external service calls, publish workflow, upload workflow, or metrics read.

## Internal-Only Token Exchange Boundary

The boundary exists so future OAuth callback work can stop at safe token exchange readiness without exchanging real tokens. It depends on Batch 2 OAuth state storage / anti-replay foundation.

Fake-gated exchange simulation can proceed only after a known provider, an obvious fake authorization code, and valid single-use state consumption are accepted. The result is metadata-only and redacted.

## Input Categories

| Input category | Expected boundary handling |
| --- | --- |
| Valid provider + valid consumed state + obvious fake authorization code | Return `exchange_simulated` with safe metadata only. |
| Missing authorization code | Return `authorization_code_missing` without consuming state. |
| Malformed authorization code | Return `authorization_code_malformed` without consuming state. |
| Missing state | Return `state_validation_failed`. |
| Malformed state | Return `state_validation_failed`. |
| Replayed state | Return `state_validation_failed`. |
| Expired state | Return `state_validation_failed`. |
| Provider mismatch | Return `state_validation_failed`. |
| Unsupported provider | Return `unsupported_provider`. |
| Real provider disabled | Return `real_provider_disabled`. |
| Sandbox fallback attempt | Return a safe blocked result; `douyin_real` must not fallback to `douyin_sandbox`. |

## Expected Safe Result Categories

- `exchange_simulated`: fake-gated exchange simulation completed after valid state consumption.
- `state_validation_failed`: Batch 2 state validation failed.
- `authorization_code_missing`: authorization code was absent.
- `authorization_code_malformed`: authorization code was not accepted for fake-gated simulation.
- `unsupported_provider`: provider id is unknown or unsupported.
- `real_provider_disabled`: `douyin_real` is disabled / blocked.
- `external_exchange_blocked`: any path that would call an external token exchange is blocked.
- `credential_storage_required`: future credential reference / encrypted storage is required before real token material can be handled.

## Redaction Rules

User-facing responses must not display authorization code, raw state, access token, refresh token, secret, or credential material.

Logs must not display authorization code, raw state, token values, secret values, credential material, cookies, sessions, raw request, or raw response payloads.

Audit records must not display authorization code, raw state, token values, secret values, credential material, cookies, sessions, raw request, or raw response payloads.

The database must not save raw authorization code, raw state, access token, refresh token, token value, secret, credential material, cookie, session, raw request, or raw response payloads.

Tests may use only obvious fake authorization code values and obvious fake sensitive values for negative assertions. Test fixtures must not use values copied from real browser URLs, provider callbacks, token responses, cookies, sessions, platform payloads, local secrets, or environment secrets.

## Dependency Gates

1. Batch 2 before state validation.
2. Batch 3 only fake-gated exchange boundary.
3. Batch 4 internal credential reference / encrypted storage boundary before credential reference handoff, with no real credential storage.
4. Batch 5 internal real provider feature flag / kill switch foundation before any later real provider runtime enablement.

No future callback route should proceed to token exchange boundary without Batch 2 state validation. No real token exchange should occur before a later accepted real exchange batch. Batch 4 can consume safe Batch 3 metadata for metadata-only credential references, and Batch 5 can guard later real provider runtime decisions, but no real token or credential material should be stored before a later accepted real storage implementation. No real provider runtime is enabled in Batch 5.

## Explicit Non-Goals

Batch 3 does not provide:

- Real OAuth.
- OAuth URL creation.
- OAuth start route.
- OAuth callback route.
- Real token exchange.
- Real token storage.
- Real credential storage.
- Frontend OAuth UI.
- External service calls.
- Real Douyin authorization.
- Real publish.
- Real metrics read.
- Upload, publish, or scheduled publishing.
- Production, commercial, or SaaS readiness.
