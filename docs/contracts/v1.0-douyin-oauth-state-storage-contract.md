# v1.0 Douyin OAuth State Storage Contract

This document defines the internal-only OAuth state storage foundation for v1.0 Batch 2. It is not a public API contract. As of Batch 3, the token exchange boundary may depend on this state foundation for fake-gated simulation, but this contract still does not expose any OAuth state API or route.

Batch 2 does not add an OAuth start endpoint, OAuth callback endpoint, OAuth URL creation, token exchange, token storage, credential storage, frontend OAuth UI, external service call, publish workflow, upload workflow, or metrics read. Batch 3 adds a separate internal-only token exchange boundary, but still does not add real token exchange, token storage, credential storage, OAuth routes, OAuth URL creation, frontend OAuth UI, external service calls, publish workflow, upload workflow, or metrics read.

## Internal-Only State Foundation Contract

The state foundation exists so a future callback route can validate OAuth state before token exchange. Batch 3 fake-gated token exchange tests can consume state through this foundation, but real token exchange still requires a later accepted implementation path. The state foundation stores only digest / hash / metadata.

Raw state may be generated internally for a future OAuth handoff, but Batch 2 must not expose raw state through API responses, frontend UI, logs, audit records, test snapshots, docs examples, or database rows.

## State Creation Contract

State creation must follow these rules:

- `provider_id` must be a known provider.
- `source_type` must match the provider registry.
- Raw state value must not be persisted.
- `state_digest` must be unique.
- `expires_at` must exist.
- `safe_status_message` must not contain sensitive values.
- Initial `state_status` must be `pending`.
- `purpose` must be `authorization`.

The safe provider sources are:

- `fake_local`
- `sandbox`
- `real`

`fake_local`, `douyin_sandbox`, and `douyin_real` must remain separated.

## State Validation / Consume Contract

Future callback validation must use the internal state foundation with these outcomes:

| Input | Expected foundation outcome |
| --- | --- |
| Valid pending state for matching provider | Mark as `consumed` and return `valid_consumed`. |
| Reused consumed state | Return `invalid_replayed_state` or equivalent invalid replay category. |
| Expired state | Mark or keep as `expired` and return `invalid_expired_state`. |
| Missing state | Return `invalid_missing_state`. |
| Malformed state | Return `invalid_malformed_state`. |
| Provider mismatch | Return `invalid_provider_mismatch`. |
| Unsupported provider | Return `unsupported_provider`. |

`douyin_real` failures or disabled behavior must not fallback to `douyin_sandbox`.

## Redaction Rules

User-facing responses must not display raw state.

Logs must not display raw state.

Audit records must not display raw state.

The database must not save raw state.

Tests may use obvious fake values only. Test fixtures must not use values copied from real browser URLs, provider callbacks, cookies, sessions, token responses, platform payloads, local secrets, or environment secrets.

## Non-Goals

Batch 2 does not provide:

- Real OAuth.
- OAuth URL creation.
- OAuth start route.
- OAuth callback route.
- Token exchange.
- Token storage.
- Credential storage.
- Frontend OAuth UI.
- External service calls.
- Real Douyin authorization.
- Real publish.
- Real metrics read.
- Upload, publish, or scheduled publishing.
- Production, commercial, or SaaS readiness.
