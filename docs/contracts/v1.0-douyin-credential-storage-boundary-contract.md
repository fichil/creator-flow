# v1.0 Douyin Credential Storage Boundary Contract

This document defines the internal-only credential reference and encrypted storage boundary for v1.0 Batch 4. It is not a public API contract.

Batch 4 does not add an OAuth start endpoint, OAuth callback endpoint, OAuth URL creation, real token exchange, real token storage, real credential storage, frontend OAuth UI, external service call, publish workflow, upload workflow, or metrics read.

## Internal-Only Boundary

The boundary exists so future OAuth and token exchange work can stop at safe credential readiness without storing real credential material. It depends on the Batch 2 OAuth state storage / anti-replay foundation and the Batch 3 token exchange boundary.

The boundary may create metadata-only credential references from safe Batch 3 token exchange metadata. It must not store raw authorization code, raw OAuth state, access token, refresh token, token value, secret, credential material, cookie, session, bearer, raw request, or raw response values.

## Input Categories

| Input category | Expected boundary handling |
| --- | --- |
| Fake-gated token exchange metadata result | Create a metadata-only credential reference or return a safe disabled category. |
| Missing token exchange result | Return `credential_storage_disabled`. |
| Malformed exchange metadata | Return `credential_storage_rejected`. |
| Attempted raw token material | Return `credential_material_rejected`. |
| Attempted raw authorization code | Return `credential_material_rejected`. |
| Attempted raw OAuth state | Return `credential_material_rejected`. |
| Unsupported provider | Return `unsupported_provider`. |
| Real provider disabled | Return `real_provider_disabled`. |
| Sandbox fallback attempt | Return a safe blocked result; `douyin_real` must not fallback to `douyin_sandbox`. |
| Encryption provider unavailable | Return `encryption_provider_unavailable` or `credential_storage_disabled`. |

## Expected Safe Result Categories

- `credential_reference_created`: metadata-only reference was created; real credential storage remains disabled.
- `credential_storage_disabled`: credential storage is disabled or required metadata is absent.
- `credential_storage_rejected`: exchange metadata was malformed or did not match the provider.
- `credential_material_rejected`: raw credential material was detected and rejected.
- `unsupported_provider`: provider id is unknown or unsupported.
- `real_provider_disabled`: `douyin_real` is disabled / blocked.
- `encryption_provider_unavailable`: no accepted encryption provider is available.
- `external_storage_blocked`: any path that would call external storage is blocked.

## Redaction Rules

User-facing responses must not display authorization code, raw state, access token, refresh token, secret, or credential material.

Logs must not display authorization code, raw state, token values, secret values, credential material, cookies, sessions, bearer values, raw request, or raw response payloads.

Audit records must not display authorization code, raw state, token values, secret values, credential material, cookies, sessions, bearer values, raw request, or raw response payloads.

Database metadata must not save raw authorization code, raw state, access token, refresh token, token value, secret, credential material, client secret, cookie, session, bearer, raw request, or raw response payloads.

Tests may use only obvious fake values for negative assertions. Test fixtures must not use values copied from real browser URLs, provider callbacks, token responses, cookies, sessions, platform payloads, local secrets, or environment secrets.

## Encryption Boundary

If no accepted encryption provider and key-management design exists, Batch 4 must not implement pseudo-encryption. Base64, hashing, XOR, ordinary encoding, and custom algorithms are not acceptable substitutes for encrypted credential storage.

Batch 4 does not add encrypted payload storage. Any encrypted-storage-required path must return a safe disabled / unavailable result and must not persist credential material.

## Dependency Gates

1. Batch 2 before state validation.
2. Batch 3 before token exchange boundary metadata.
3. Batch 4 only credential reference / encrypted storage boundary.
4. Batch 5 before real provider feature flag / kill switch runtime enablement.

No real credential persistence should occur before a later accepted real storage implementation and real-provider enablement path. No real provider runtime should be enabled before Batch 5 controls.

## Explicit Non-Goals

Batch 4 does not provide:

- Real OAuth.
- OAuth URL creation.
- OAuth start route.
- OAuth callback route.
- Real token exchange.
- Real token storage.
- Real credential storage.
- Frontend OAuth UI.
- External service calls.
- Real Douyin authorization.
- Real publish.
- Real metrics read.
- Upload, publish, or scheduled publishing.
- Production, commercial, or SaaS readiness.
