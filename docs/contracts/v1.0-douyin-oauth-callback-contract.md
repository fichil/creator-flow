# v1.0 Douyin OAuth Callback Contract

This document defines the future callback contract for the v1.0 Douyin Integration User Test Release. It is a contract and boundary document only. It does not implement a backend route, frontend UI, OAuth URL, token exchange, token storage, credential storage, provider runtime behavior, or external service call. Batch 2 adds a separate internal-only state foundation in [`v1.0-douyin-oauth-state-storage-contract.md`](v1.0-douyin-oauth-state-storage-contract.md), but this callback contract still does not make any callback route callable.

## Future Endpoint Boundary

Future implementation may introduce an OAuth authorization start endpoint and an OAuth callback endpoint only after the required dependency gates are accepted.

Possible future route shapes may include:

- Authorization start: `GET /api/provider-auth/{provider_id}/oauth/start`
- Callback: `GET /api/provider-auth/{provider_id}/oauth/callback`

These route shapes are future contract examples, not current routes. Batch 1 did not add them, and Batch 2 still does not add them or make them callable.

## Callback Input Categories

Future callback handling must classify inputs into safe categories before any token exchange or persistence:

| Input category | Required future handling |
| --- | --- |
| Success callback with `code` and `state` | Validate provider, feature flag, callback policy, state existence, state freshness, state single-use, redirect URI policy, and redaction before returning `pending_token_exchange`. |
| User denied / cancelled | Treat as user-controlled cancellation and return `authorization_cancelled` without token exchange or credential storage. |
| Provider error | Return `authorization_failed` with a safe error class and no raw provider payload. |
| Malformed callback | Return `invalid_callback` and redact all untrusted inputs. |
| Missing state | Return `invalid_callback`; a callback without state cannot proceed. |
| Replayed state | Return `invalid_callback`; replayed state cannot proceed. |
| Expired state | Return `invalid_callback`; expired state cannot proceed. |
| Unsupported provider | Return `unsupported_provider`; do not fallback to sandbox or fake providers. |

## Expected Safe Response Categories

Future route responses must use safe categories rather than raw provider values:

- `pending_token_exchange`: state was valid and the callback is ready for the later Batch 3 token exchange boundary.
- `authorization_cancelled`: the user denied or cancelled authorization.
- `authorization_failed`: the provider returned an error or the platform rejected the authorization attempt.
- `invalid_callback`: the callback was malformed, missing state, replayed, expired, or otherwise failed validation.
- `unsupported_provider`: the provider id is not supported for this OAuth flow.
- `oauth_temporarily_disabled`: real OAuth is disabled by feature flag or kill switch.

These categories are contract outcomes only. They do not imply that any runtime route exists today.

## Redaction Rules

User-facing responses must not display:

- Authorization code values.
- OAuth state values.
- Token values.
- Secret values.
- Credential material.
- Cookie or session values.
- Raw provider request or response payloads.

Logs, audit records, error payloads, screenshots, test snapshots, and fixtures must not include:

- Real access tokens.
- Real refresh tokens.
- Real token values.
- Real client secrets.
- Real API keys.
- Real authorization codes.
- Real OAuth state values.
- Real cookies or sessions.
- Real credential material.
- Raw provider request or response payloads.

Tests may use only obvious fake values, for example `fake_code_for_contract_test`, `fake_state_for_contract_test`, `fake_provider_error`, and `fake_request_id`. Test values must never be copied from a real Douyin callback, browser URL, log, cookie, session, token response, or provider payload.

## Dependency Gates

Future implementation must follow this order:

1. Batch 2 state foundation before any callback route implementation.
2. Batch 3 before token exchange.
3. Batch 4 before credential storage.
4. Batch 5 before real feature flag / kill switch runtime enablement.

Batch 2 now provides internal state validation primitives, but no callback route exists in Batch 2. A future callback route must use Batch 2 state validation, must not proceed without single-use state validation, must not perform token exchange before Batch 3, must not store token or credential material before Batch 4, and must not enable a real OAuth entry point for testers before Batch 5.

## Provider Isolation

`fake_local`, `douyin_sandbox`, and `douyin_real` must remain strictly separated:

- `fake_local` does not require OAuth and cannot be used as a real provider callback target.
- `douyin_sandbox` may be used only for sandbox/mock contract tests and must not be represented as real Douyin OAuth.
- `douyin_real` is the future real provider path and must not fallback to `douyin_sandbox`.
- Unknown or unsupported provider ids must fail with `unsupported_provider`.

## Explicit Non-Goals

Batch 2 still does not provide:

- Real OAuth.
- OAuth URL creation.
- OAuth callback route.
- Token exchange.
- Token storage.
- Credential storage.
- External service calls.
- Database changes.
- Frontend OAuth UI.
- Real Douyin authorization.
- Real publish.
- Real metrics read.
- Upload, publish, or scheduled publishing.
- Production, commercial, or SaaS readiness.
